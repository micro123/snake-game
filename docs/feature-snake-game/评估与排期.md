# 评估与排期

> **功能**: 经典贪吃蛇游戏 (Snake Game) | **阶段**: 评估与排期 (P4，子步骤2) | **角色**: tech-lead + project-manager | **日期**: 2026-07-07

---

## 1. 复杂度评估

### 1.1 模块复杂度

| 模块 | 难度(1-5) | 说明 |
|------|----------|------|
| config | 1 | 纯常量定义，无逻辑，零复杂度。直接映射设计文档配置值。 |
| snake | 3 | 核心游戏逻辑：move_and_grow 合二为一、反向拦截、边界碰撞(坐标范围检查)、自撞检测(head in body[1:])。尾部弹出逻辑和 (-dx,-dy) 拒绝是关键正确性保证点。 |
| food | 2 | 简单逻辑：计算所有格子与占用格子的集合差，随机选取。处理棋盘满(返回 False 触发胜利)和初始占位 (-1,-1)。无算法复杂度，但必须保证食物从不出现在蛇身上。 |
| renderer | 3 | 封装所有 Pygame 绘制调用：分层渲染（背景->网格线->蛇身->蛇头->食物->HUD->叠加层）、蛇头蛇身颜色区分、半透明游戏结束/胜利叠加层及居中文字。管理 Surface、字体、Rect 定位、Clock 帧率控制。Pygame API 熟悉度影响开发速度。 |
| input_handler | 2 | Pygame 事件队列遍历、方向键到方向向量映射、每帧单方向缓冲（最后有效键胜出）、焦点丢失/恢复检测实现自动暂停、方向反转拦截委托给 Snake.change_direction。事件循环模式中等复杂度但直接了当。 |
| game | 2 | 编排层：连接各组件形成主循环 (tick -> process -> handle -> update -> render -> flip)。实现 RUNNING/GAME_OVER/VICTORY 三态状态机。处理 R 键重置所有组件。无复杂算法但循环步骤顺序至关重要。 |
| main | 1 | 入口点：try/except ImportError 检查 Pygame、pygame.init()、Game 实例化、run()、pygame.quit() 于 finally。约 5-8 行代码。 |

### 1.2 综合评估

| 维度 | 评级 |
|------|------|
| 整体复杂度 | 中等 (medium) |
| 工作量估算 | 2-3 人天 (min:2, max:3) |
| 优先级 | P3（仓库唯一功能，自包含无竞争依赖） |

**理由**: 经典贪吃蛇游戏，逻辑已被充分理解。设计文档提供了完整的类接口、方法签名、数据流和 6 条关键流程序列——实现是规格驱动的。7 个模块中无一达到高复杂度；snake (3/5) 和 renderer (3/5) 因精度要求而非算法难度成为最具挑战性的模块。整个功能适配单次中小型迭代。

**建议构建方式**: 自底向上构建：(1) config 常量，(2) Snake 类及单元测试覆盖移动/增长/碰撞，(3) Food 类及重生逻辑，(4) Renderer 通过 mock Surface 验证，(5) InputHandler 模拟事件测试，(6) Game 协调器连接全部组件，(7) main 入口点。测试重点：碰撞边界情况（单节蛇、棋盘边缘转向、快速连按）、接近满棋盘的食物生成、以及重启状态一致性。现有 main.py 骨架可废弃或重构，设计的模块化架构完全替代之。

**运行时依赖**:
- Python 3.x 运行时
- Pygame 库 (pip install pygame)

## 2. 风险评估

| 风险 | 类型 | 严重度 | 描述 | 缓解措施 |
|------|------|--------|------|----------|
| R-001 | 依赖 | 低 | Pygame 未安装 — ImportError at startup | main.py 捕获 ImportError 输出清晰 pip install 提示并 sys.exit(1)。README 说明前置要求。 |
| R-002 | 技术 | 低 | 焦点丢失事件 (WINDOWEVENT_FOCUS_LOST) 部分平台不支持 | 优雅降级：_paused 保持 False，游戏继续不会崩溃。可接受的体验降级。 |
| R-003 | 技术 | 低 | 自撞时序：移动前 vs 移动后检查可能引发误判或漏判 | 设计指定先移后检顺序。head 插入 body[0] 后对 body[1:] 检查碰撞——规格正确。 |
| R-004 | 性能 | 低 | 棋盘接近满时食物重生成：随机采样小剩余集可能循环 | 设计用集合差方法 (all_cells - occupied) 而非重试循环——O(n) 保证，无随机最差情况。 |

## 5. 实施顺序

### Phase 1: 基础设施 (0.5h)
- **任务**: T-001
- **理由**: config.py 是所有模块的基础依赖，定义全部共享常量。必须最先完成，后续模块才能引用 COLORS、GRID_COLS/ROWS、FPS 等值。零复杂度，立即产出。

### Phase 2: 核心逻辑层 (6h，可并行)
- **任务**: T-002, T-003
- **理由**: Snake 和 Food 均为纯逻辑组件，无 Pygame 依赖。两者依赖关系独立（都仅依赖 T-001 config），可以并行开发。Snake 的碰撞检测是游戏核心，优先完成以便充分验证；Food 的集合差生成算法是胜利条件的唯一判定点。两者完成即可进行逻辑层单元测试，无需图形环境。

### Phase 3: 表现与输入层 (6h，可并行)
- **任务**: T-004, T-005
- **理由**: Renderer 需要 T-002 (Snake) 和 T-003 (Food) 的类定义来遍历 body 和读取 position，但无需完整的 game 集成。InputHandler 仅依赖 T-001 config，与前两者无直接依赖。两者可作为独立的 I/O 层并行实现。Renderer 可用模拟数据进行视觉验证，InputHandler 可用模拟事件队列进行单元测试。

### Phase 4: 集成协调层 (3h)
- **任务**: T-006
- **理由**: Game 是协调器，依赖全部 5 个下层模块。在 Phase 1-3 全部就绪后才能正确连接。主循环的步骤顺序 (`tick -> process -> handle -> update -> render -> flip`) 必须严格按照设计文档编排，错误顺序会导致状态不一致。三态状态机 (RUNNING/GAME_OVER/VICTORY) 的迁移逻辑在此一次性实现。

### Phase 5: 入口与发布 (2h，可部分并行)
- **任务**: T-007, T-008
- **理由**: main.py 依赖 T-006 Game 类，需在集成完成后编写。README 与代码无关，可与 T-007 并行，或在 T-007 完成后同步交付。两者都是发布阶段的收尾工作。

### Phase 6: 质量验证 (3h)
- **任务**: T-009
- **理由**: 全部代码完成后进行系统性验收测试。必须覆盖 9 条验收标准 (AC)、9 个边界情况 (E-001~E-009)、4 项风险评估的缓解验证。这是发现集成缺陷和边界遗漏的最后防线，必须在所有功能代码完成后执行。
