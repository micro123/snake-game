# 需求规格文档

> **功能**: 方向键输入缓冲系统 (input-buffering) | **阶段**: 需求分析 (P1，子步骤2) | **角色**: 需求工程师 | **日期**: 2026-07-11

---

## 1. 功能性需求

### FR-001 方向输入 FIFO 缓冲队列

- **描述**: 在 InputHandler 中维护一个固定容量为 4 的 FIFO 方向缓冲队列。KEYDOWN 方向键事件由 Game._handle_command() 统一拦截后转发至 InputHandler.enqueue_direction() 入队（而非直接变更蛇方向）。每逻辑 tick 由 Game._update() 调用 InputHandler.consume_direction() 至多出队 1 个方向并提交给蛇。
- **优先级**: P0
- **验收标准**:
  - AC-1.1: 队列采用 collections.deque 实现，最大长度 maxlen=4
  - AC-1.2: 新方向追加到队尾；队列满时最早条目自动被丢弃（FIFO 溢出策略）
  - AC-1.3: 每逻辑 tick 至多出队 1 个方向；队列为空时该 tick 无方向提交
  - AC-1.4: 队列存储在 InputHandler 实例中，游戏 reset() 时重新初始化
  - AC-1.5: 方向入队入口为 InputHandler.enqueue_direction()，由 Game._handle_command() 在 KEYDOWN 方向键分支中调用
  - AC-1.6: Snake 不持有 InputHandler 引用，不保留 Snake.change_direction() 方法
  - AC-1.7: 在单次逻辑 tick 间隔内依次按下 UP、LEFT、DOWN 三个方向键，蛇在后续三个连续的逻辑 tick 中各执行一个方向变更（先 UP、再 LEFT、再 DOWN），而非仅执行最后的 DOWN

### FR-002 入队前校验与去重

- **描述**: 方向入队前做基本有效性校验（是否属于上下左右四方向之一），并对队尾连续同向做去重（若新方向与队尾方向相同则跳过入队）。队空时以蛇当前 committed_direction 为参照执行反向过滤——与 committed_direction 相同或相反的方向拒绝入队，防止玩家误操作导致无效方向（原地不动）或自杀方向入队。蛇当前向右移动时玩家按 LEFT 再按 RIGHT：LEFT 与 committed_direction(RIGHT) 相反被拒绝，RIGHT 与 committed_direction 相同被拒绝，队列保持为空。
- **优先级**: P0
- **验收标准**:
  - AC-2.1: 仅当新方向与队尾方向不同且非 suicide（不与队尾相反）时才追加到队列
  - AC-2.2: 有效性校验依据现有 DIRECTION_MAP 进行，非上下左右的方向键不进入缓冲
  - AC-2.3: 连续快速按下相同方向键（如两次 RIGHT），队列中仅保留一次 RIGHT
  - AC-2.4: 蛇向右移动时队列中有 UP 后玩家按下 DOWN：DOWN 与队尾 UP 相反被拒绝；队列仅保留 UP
  - AC-2.5: 入队阶段执行反向过滤——与队尾相反的方向被拒绝（队空时以 committed_direction 为参照）
  - AC-2.6: 队尾去重不影响非连续同向场景（如右→下→右 应全部入队，第一次右入队，下入队，第二次右入队）
  - AC-2.7: 蛇当前向上移动（committed_direction=UP），玩家快速依次按下 UP、LEFT、DOWN：第一个 UP 因队空时与 committed_direction 相同被去重拒绝，仅 LEFT 和 DOWN 入队为 [LEFT, DOWN]，而非三个键全部入队。即 180 度掉头由 LEFT+DOWN 两键实现，非 UP+LEFT+DOWN 三键

### FR-003 每逻辑 tick 消费一个方向并应用到蛇

- **描述**: 每个逻辑 tick 从队列头部消费一个方向，若方向有效则应用到蛇（写入 snake.direction），替代当前逐渲染帧立即改向的机制。若方向无效或队列为空则不改变蛇的方向。消费发生在 Snake.move_and_grow() 调用之前。
- **优先级**: P0
- **验收标准**:
  - AC-3.1: consume_direction() 方法每次调用至多返回 1 个有效方向或 None
  - AC-3.2: 消费速率与逻辑 tick 绑定（tick_interval 间隔），不受渲染帧率影响
  - AC-3.3: 在加速/高难度状态下逻辑 tick 间隔缩短时速率限制仍然生效
  - AC-3.4: 若队列为空则不改变蛇的方向

### FR-004 禁用操作系统级按键重复

- **描述**: 在 main.py 的 Pygame 初始化阶段显式调用 `pygame.key.set_repeat(0)`，确保跨平台行为一致，防止 OS 向事件队列注入重复 KEYDOWN 事件破坏缓冲的确定性。
- **优先级**: P0
- **验收标准**:
  - AC-4.1: `pygame.key.set_repeat(0)` 在 `pygame.init()` 之后、主循环之前调用
  - AC-4.2: 该调用在 Linux、Windows、macOS 上均生效
  - AC-4.3: 队尾去重作为 OS 级重复未生效时的防御性兜底

### FR-005 缓冲队列 HUD 可视化显示

- **描述**: 在 HUD 上直观显示当前缓冲队列中的方向序列，使用方向箭头符号（^ v < >）排列在现有 HUD 信息区域。空白队列时不显示或显示为空标识，确认玩家的快速输入已被系统正确接收。
- **优先级**: P2
- **验收标准**:
  - AC-5.1: HUD 在游戏区域顶部合适位置显示队列内容，使用方向箭头符号（^ v < >）标示当前排队的移动方向序列
  - AC-5.2: 空白队列时不显示或显示为空标识
  - AC-5.3: 队列内容实时更新，每次渲染帧反映最新缓冲状态
  - AC-5.4: 渲染逻辑扩展自 Renderer.draw_hud()

### FR-006 队列清空策略

- **描述**: 游戏重置（restart / R 键）时清空缓冲队列，确保上一局遗留的缓冲不影响新对局。游戏结束（game_over）、胜利（victory）、焦点丢失（暂停）时不清空队列——队列在非 RUNNING 状态下保留但不消费，玩家输入保留至下次 reset() 时统一清空。
- **优先级**: P0
- **验收标准**:
  - AC-6.1: reset() 中调用 input_handler.flush_queue() 清空队列
  - AC-6.2: game_over 或 victory 状态进入时不清空队列，队列保留至下次 reset() 时清空
  - AC-6.3: 焦点丢失导致暂停时不清空队列，队列保留至下次 reset() 时清空
  - AC-6.4: 按下 R 键 restart 后缓冲队列被清空
  - AC-6.5: flush_queue() 方法以 queue.clear() 实现

### FR-007 方向输入统一拦截与循环依赖断开

- **描述**: 方向输入由 Game._handle_command() 统一拦截 KEYDOWN 方向键事件，转发至 InputHandler.enqueue_direction() 入队。Snake 不持有 InputHandler 引用，删除 Snake.change_direction() 方法。input_handler.py 从 config 导入 GameState（而非从 game 导入），断开既有 `game -> input_handler -> game` 循环导入链。
- **优先级**: P0
- **验收标准**:
  - AC-7.1: `input_handler.py` 中 `from game import GameState` 改为 `from config import GameState`
  - AC-7.2: `Game._handle_command()` 中方向键分支调用 `self.input_handler.enqueue_direction(dir)`
  - AC-7.3: Snake 类中删除 `change_direction()` 方法
  - AC-7.4: import 链路无循环：config <- input_handler, config <- snake, game -> {input_handler, snake, config}

### FR-008 测试覆盖需求

- **描述**: 方向输入缓冲系统的测试需覆盖新增队列逻辑及既有功能变更的影响。InputHandler 单元测试覆盖入队（校验/去重/反向过滤/溢出）、出队（有效方向返回/反向跳过/空队列）、清空。集成测试适配 process_events 返回格式变更（direction 键移除）及 Snake.change_direction() 删除后的行为。实施顺序建议：先编写 InputHandler 单元测试，再适配集成测试。
- **优先级**: P1
- **验收标准**:
  - AC-8.1: InputHandler 单元测试覆盖 enqueue_direction 入队逻辑：有效方向入队、队尾同向去重、队尾反向拒绝、队空时以 committed_direction 为参照的反向拒绝、FIFO 溢出丢弃
  - AC-8.2: InputHandler 单元测试覆盖 consume_direction 出队逻辑：有效方向出队、连续反向条目跳过、空队列返回 None
  - AC-8.3: InputHandler 单元测试覆盖 flush_queue 清空和 get_queue_snapshot 快照
  - AC-8.4: 集成测试中 process_events 返回格式不再包含 direction 键，原 `command['direction']` 断言迁移为队列状态断言
  - AC-8.5: 原 Snake.change_direction() 的 ~19 条测试用例迁移至 InputHandler 层（enqueue_direction + consume_direction 组合行为）
  - AC-8.6: 集成测试中额外 ~10 条 command['direction'] 相关断言经审查后迁移为队列消费行为验证

## 2. 非功能性需求

### 2.1 性能

- NFR-P01: 缓冲入队/出队/去重/清空操作均为 O(1) 时间复杂度
- NFR-P02: 每逻辑 tick 缓冲操作额外开销不超过 100us，对逻辑帧率无可观测影响
- NFR-P03: pygame.key.set_repeat(0) 调用在初始化阶段执行一次，运行时无额外开销
- NFR-P04: HUD 队列可视化渲染开销不显著影响帧率（仅渲染至多 4 个箭头字符）

### 2.2 安全

- NFR-S01: 本地单玩家游戏，无网络通信或用户数据持久化，不涉及安全需求
- NFR-S02: 方向输入缓冲不引入新的数据注入或篡改风险（输入仅来自 Pygame 事件系统）

### 2.3 可用性

- NFR-U01: 方向缓冲行为对玩家完全透明——快速连续按键自然得到正确响应，无需学习新操作方式
- NFR-U02: HUD 可视化提供直观的队列状态反馈，增强玩家对输入系统的信任感
- NFR-U03: 按住方向键不产生意外的连续转向（OS 级重复已禁用 + 队尾去重双重保障）
- NFR-U04: 队列自动排除与队尾相同或相反的方向，防止误操作导致无效或自杀方向

### 2.4 可靠性

- NFR-R01: 队列最大长度硬编码为 4，确保高操作密度下缓冲永不无限增长
- NFR-R02: 队列溢出采用 FIFO 丢弃策略，丢弃最早而非最新方向，符合玩家直觉
- NFR-R03: 仅 reset() 时清空队列；game_over/victory/暂停不清空，避免静默丢弃玩家已输入但尚未执行的方向
- NFR-R04: 循环依赖已断开，无双向导入，模块结构清晰可维护
- NFR-R05: 队尾同向去重和 OS 级重复禁用双重保障，确保各平台行为一致

## 3. 约束条件

- C-01: 依赖 Pygame 事件系统及 pygame.key.set_repeat() API
- C-02: InputHandler 与 Snake 运行在同一进程/线程中，无并发安全问题
- C-03: 队列长度固定硬编码为 4，不提供用户可配置项
- C-04: 仅处理上下左右四方向（依据现有 DIRECTION_MAP），R/Q/空格/回车等不进入缓冲
- C-05: 目标平台为 Linux、Windows、macOS（pygame 支持的三个主要桌面平台）
- C-06: InputHandler 从 config 导入 GameState（不从 game 导入），避免循环依赖
- C-07: Snake 不持有 InputHandler 引用，方向变更仅通过 Game 层统一管理
- C-08: committed_direction 属性仅在 Snake.move_and_grow() 执行时更新
- C-09: 方向键以外按键（R/Q/空格）不进入缓冲队列
- C-10: 触摸屏手势或鼠标输入不在缓冲范围内
- C-11: 不包含输入预测或平滑算法
- C-12: 队列无持久化存储或跨会话恢复能力
- C-13: 加速键不影响队列行为，仅影响逻辑 tick 速率

## 4. 数据实体

### 4.1 DirectionQueue

- **存储位置**: InputHandler 实例属性
- **字段**:
  - `queue`: `collections.deque` — 方向 FIFO 队列，maxlen=4，存储方向向量元组 (dx, dy)
- **关联关系**:
  - InputHandler 持有 DirectionQueue 作为实例属性（组合关系）
  - 队列中每条目与 Snake.committed_direction 同类型
  - 入队由 Game._handle_command() 触发 InputHandler.enqueue_direction()
  - 出队由 Game._update() 触发 InputHandler.consume_direction()
  - 清空由 flush_queue() 方法执行，仅在 reset() 时调用

### 4.2 Direction（既有枚举/常量）

- **字段**:
  - 枚举值: UP, DOWN, LEFT, RIGHT — 四方向值，各携带 (dx, dy) 向量元组
- **关联关系**:
  - 在 DirectionQueue.queue 中作为条目存储
  - 在 Snake.committed_direction 中作为当前物理方向引用
  - 在入队反向过滤和出队校验中作为参数参与判断

### 4.3 GameState（既有枚举；从 config 导入）

- **字段**:
  - 枚举值: RUNNING, PAUSED, GAME_OVER, VICTORY, MENU
- **关联关系**:
  - InputHandler 从 config 导入 GameState，不依赖 game 模块
  - Game._update() 依据 GameState 判断是否执行方向消费：仅 RUNNING 状态下消费队列
  - GameState 变更为非 RUNNING（PAUSED/GAME_OVER/VICTORY）时不清空队列，队列保留至 reset() 时由 flush_queue() 清空

## 5. 边界情况

| # | 场景 | 期望行为 |
|---|------|----------|
| 1 | 队列已满（4 条方向）且新方向入队 | 最早条目被 FIFO 丢弃，新方向追加到队尾。丢弃的是最旧方向而非最新方向 |
| 2 | 队列为空时调用 consume_direction() | 返回 None，蛇方向不变，该 tick 无转向 |
| 3 | 出队时队列中所有方向均为反向 | 全部丢弃，该 tick 无方向提交，蛇保持当前方向继续移动 |
| 4 | 同一方向非连续多次输入（如右→下→右） | 第二次右不属于队尾同向（队尾为下），应正常入队 |
| 5 | 玩家在暂停状态按下方向键 | 暂停时不清空队列，方向键仍可入队；恢复 RUNNING 后缓冲中的方向按序消费 |
| 6 | 游戏结束瞬间缓冲中仍有未消费方向 | GameOver 时不清空队列，残留方向保留至下次 reset() 时清空；非 RUNNING 状态下队列不消费 |
| 7 | pygame.key.set_repeat(0) 未生效的极端平台 | 队尾去重作为防御性兜底，即使 OS 注入重复事件也不会填满队列 |
| 8 | 逻辑 tick 间隔短于按键事件到达间隔（高难度 20ms tick） | 队列最多缓 4 tick 预输入（80ms 窗口），超出部分丢弃 |
| 9 | 蛇初始方向向上（committed_direction=UP），玩家快速输入 UP、LEFT、DOWN 三键以完成 180 度掉头 | **UP 被拒绝**：队空时以 committed_direction(UP) 为参照，UP 与参照相同，被去重拒绝不入队；LEFT 入队（与队尾不存在、与参照 UP 非相反）；DOWN 入队（与队尾 LEFT 非相反、非相同）。最终队列为 [LEFT, DOWN]。tick1 出队 LEFT，tick2 出队 DOWN，实现 180 度掉头。**关键澄清**：此场景下仅 LEFT 和 DOWN 两键入队，UP 不入队；180 度掉头并非由 UP+LEFT+DOWN 三键实现，而是 LEFT+DOWN 两键实现 |
| 10 | 两个连续入队的方向恰好互为反向 | 第一个消费后蛇转向，第二个在下一 tick 消费时与新的当前方向相反，被正确拦截丢弃 |
| 11 | 玩家在单帧内产生超过 4 个有效方向输入 | 队列 FIFO 溢出，后入方向覆盖先入方向 |
| 12 | victory 状态时缓冲中有未消费方向 | victory 时不清空队列，残留方向保留至下次 reset() 时清空；非 RUNNING 状态下队列不消费 |
| 13 | 焦点丢失导致游戏暂停 | 队列不清空，暂停期间不消费方向；恢复 RUNNING 后按序消费；reset() 时清空 |

## 7. 假设条件

1. 假设 pygame.key.set_repeat(0) 在所有目标平台（Linux/Windows/macOS）上均生效
2. 假设逻辑 tick 间隔在极端难度下不低于 20ms（MIN_TICK_INTERVAL 硬下限），队列长度 4 足以覆盖 4 tick 即 80ms 的预输入窗口
3. 假设玩家不会在单 tick 内产生超过 4 个有效方向输入（正常人手速极限约 10-15 次/秒）
4. 假设 InputHandler 与 Snake 运行在同一进程/线程中，无并发安全问题
5. 假设现有 Snake.committed_direction 的值始终有效，且仅在 move_and_grow 时更新；Snake.direction 在外部写入时确保为有效方向值
6. 假设 game_over 和 victory 状态下仍可按下 R/Q，但这些按键本身不经过方向队列——仅方向键队列在非 RUNNING 状态下保留但不消费，reset 时清空
7. 假设 HUD 队列可视化使用方向箭头符号（^ v < >）表示队列中每个方向，排列在现有 HUD 信息区域
8. 假设加速键不影响队列行为，仅影响逻辑 tick 速率
9. 假设方向键在非 RUNNING 状态下仍可通过事件系统入队，但队列仅在 RUNNING 状态下被消费
10. 假设玩家不使用宏或脚本工具模拟超人类手速的连续输入；正常人类输入速率在队列容量 4 的设计范围内，宏/脚本场景不作为设计目标
