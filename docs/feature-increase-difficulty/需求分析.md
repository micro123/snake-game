# 需求规格文档

> **功能**: 分数递增难度（Score-Based Difficulty Scaling） | **阶段**: 需求与设计 | **角色**: requirements-engineer | **日期**: 2026-07-10

## 1. 问题陈述

当前游戏蛇的移动速度固定为基准 tick 间隔 100ms，无论玩家吃多少食物、拿多少分，难度始终不变。这导致熟练玩家在后期缺乏挑战，游戏体验趋于单调。需要建立分数与蛇移动速度之间的关联：分数越高，蛇移动越快，游戏自然变难，从而持续提供挑战和成就感。

## 2. 用户故事

用户故事是本文档的全文锚点。所有后续功能需求、非功能需求、边界情况和验收标准均围绕以下故事展开。

| 角色 | 我想要 | 以便 | 优先级 |
|------|--------|------|--------|
| 玩家 | 蛇的移动速度随分数增长而自动提高 | 游戏越玩越有挑战，不会感到单调 | must-have |
| 玩家 | 难度递增过程平滑且可感知 | 我能适应逐渐变快的节奏，不至于被突然加速吓到 | must-have |
| 玩家 | 每局开始时难度从初始水平重新计算 | 每次重新开始都是公平的起点 | must-have |
| 玩家 | 手动加速键（空格）与分数驱动加速叠加生效 | 我在任意分数段仍可主动获取额外速度 | should-have |
| 玩家 | 在界面上看到当前难度等级 | 我能直观了解当前处于什么难度阶段 | should-have |

## 3. 目标用户

贪吃蛇游戏的全体玩家（单机单用户），尤其是有一定操作基础、希望在长局中获得递增挑战的玩家。

## 4. 范围边界

### 范围内

- 基于分数的自动加速倍率计算逻辑（在现有 tick interval 系统中新增一个难度倍率因子）
- 难度倍率与手动 boost 倍率的协同叠加机制
- 分数阈值-倍率映射表的配置常量（例如每 50 分 +0.1x）
- 难度等级/倍率的 HUD 显示（例如 "Lv.3  Speed 1.3x"）
- 游戏重启时难度倍率复位到初始值 1.0

### 范围外

- 独立的难度选择菜单（简单/普通/困难模式）—— 不在本次范围
- 难度影响游戏元素的变化（如障碍物增减、棋盘缩小、食物出现概率等）—— 仅限速度
- 蛇身颜色随难度变化的视觉反馈 —— 仅限 HUD 文字指示
- 难度参数的持久化存储（关闭游戏后记忆）—— 仅限单局生命周期
- 在移动速度之外引入其他难度维度（如双蛇、反向控制等）

## 5. 验收标准

- 基础 tick 间隔随分数增长而缩短，分数越高蛇移动越快
- 分数 0 时 tick 间隔 = 100ms（现有基准），保持不变
- tick 间隔存在可配置的下限（默认 20ms），防止无限加速
- 难度等级在游戏重启（reset）时归零
- 手动加速（空格键）与分数驱动加速互不冲突，叠加后可接近但不低于 tick 下限
- 分数阈值与加速倍率的映射关系可通过配置常量调整

## 6. 功能性需求

### FR-001 分数驱动的难度倍率计算

**描述**：根据当前分数和预设阈值-倍率映射表，计算出当前 difficulty_multiplier。分数 0 时 difficulty_multiplier = 0（即基础速度 1.0x），随分数跨越阈值台阶式递增。

**优先级**：P0

**验收标准**：
- 分数 0 时 tick 间隔 = 100ms（现有基准不变）
- 分数每跨越一个阈值（默认每 50 分），difficulty_multiplier 累加固定增量（默认 +0.1）
- tick 间隔 = BASE_TICK_INTERVAL / (1 + difficulty_multiplier + boost_multiplier)，不得低于 MIN_TICK_INTERVAL（20ms）
- 分数阈值和增量值通过配置常量定义，修改常量后行为随之改变

**回应用户故事**：蛇的移动速度随分数增长而自动提高（故事 1）；难度递增过程平滑且可感知（故事 2）。

---

### FR-002 难度倍率与手动 boost 倍率的叠加

**描述**：玩家按住空格键产生的 boost_multiplier 与 FR-001 的 difficulty_multiplier 以加法叠加，共同影响最终 tick 间隔。两套倍率互不干扰，各自独立计算。

**优先级**：P0

**验收标准**：
- 同时存在 difficulty_multiplier 和 boost_multiplier 时，综合倍率 = 1 + difficulty_multiplier + boost_multiplier
- 仅分数加速无手动 boost 时，tick 间隔 = BASE / (1 + difficulty_multiplier)，行为正常
- 仅手动 boost 无分数加速时，tick 间隔 = BASE / (1 + boost_multiplier)，行为与改动前一致
- 两者同时最大时，tick 间隔不低于 20ms 硬下限

**回应用户故事**：手动加速键与分数驱动加速叠加生效（故事 4）。

---

### FR-003 难度状态随游戏重启复位

**描述**：调用 Game.reset() 时，difficulty_multiplier 和内部难度等级计数器全部归零，确保每局从相同初始难度开始。

**优先级**：P0

**验收标准**：
- reset() 后 difficulty_multiplier = 0
- reset() 后 tick 间隔恢复为 100ms（无 boost 状态下）
- reset() 后 HUD 难度显示恢复为初始等级
- 连续多次 reset() 行为一致

**回应用户故事**：每局开始时难度从初始水平重新计算（故事 3）。

---

### FR-004 难度等级 HUD 显示

**描述**：在游戏画面渲染时，新增一行 HUD 文字显示当前难度等级和综合倍率，格式如 "Lv.3  Speed 1.3x"。等级基于分数阈值区间计算，倍率为当前实际生效的综合倍率（含 boost）。

**优先级**：P1

**验收标准**：
- HUD 新增一行显示难度等级编号和综合速度倍率
- 难度等级随分数跨越阈值实时更新
- 综合倍率在 boost 激活/释放时实时反映变化
- 显示内容不遮挡蛇、食物等游戏元素
- 文字清晰可读，与现有 HUD 风格一致

**回应用户故事**：在界面上看到当前难度等级（故事 5）。

---

### FR-005 难度参数配置化

**描述**：所有难度相关参数（分数阈值间隔、每级倍率增量、最大倍率上限、最小 tick 间隔）定义在 config.py 中作为模块级常量，核心逻辑从配置读取而非硬编码。

**优先级**：P1

**验收标准**：
- config.py 中定义 SCORE_THRESHOLD_INTERVAL（默认 50）
- config.py 中定义 DIFFICULTY_INCREMENT（默认 0.1）
- config.py 中定义 MAX_DIFFICULTY_MULTIPLIER（默认计算值，使 BASE/(1+max_diff) >= MIN_TICK_INTERVAL）
- 修改配置常量后重启游戏即生效

**回应用户故事**：分数阈值与加速倍率映射关系可调（故事 1 的可配置性支撑）。

---

### FR-006 tick 间隔下限强制保护

**描述**：无论 difficulty_multiplier 和 boost_multiplier 如何叠加，最终计算出的 tick_interval 不得低于 MIN_TICK_INTERVAL（20ms），防止无限加速导致游戏不可玩。

**优先级**：P0

**验收标准**：
- 任意分数 + 任意 boost 状态下，实际 tick 间隔 >= 20ms
- 接近下限时加速效果自然收窄，不出现跳跃式截断
- 到达下限后继续吃食物不导致崩溃或异常

**回应用户故事**：难度递增可感知但不致失控（故事 2 的安全保障）。

## 7. 非功能性需求

### 7.1 性能

- 难度倍率计算在每次 tick 间隔查询时执行（当前约 10 次/秒），计算复杂度 O(1)，不引入可测量的帧时间开销
- HUD 难度文字渲染复用现有 Renderer 字体缓存机制，不额外创建 Font 对象
- 难度参数从 config 模块级常量读取，无 IO 或动态计算开销

### 7.2 安全

- 本功能为纯客户端单机游戏，无安全敏感数据
- 难度参数均为本地配置常量，不涉及外部输入验证

### 7.3 可用性

- 难度递增为台阶式（每 50 分一跳），每次变化幅度固定（0.1x），玩家可清晰感知等级提升
- HUD 难度显示位于屏幕角落或状态区，不干扰游戏主视野
- 初始难度与改动前体验一致（100ms tick），对新手无负面影响
- 加速叠加后 tick 间隔不短于 20ms（50 tick/s），确保游戏始终可操作

### 7.4 可靠性

- tick 间隔计算使用 max() 硬限保护，不会因浮点误差或异常参数跌破下限
- difficulty_multiplier 使用浮点数累加，精度范围内稳定（0.1 累加 100 次误差可忽略）
- reset() 覆盖所有难度状态变量，不会残留上一局数据

## 8. 数据实体

### 8.1 DifficultyState

| 字段 | 类型 | 说明 |
|------|------|------|
| difficulty_multiplier | float | 当前难度倍率累加值，初始 0.0，每跨阈值递增（默认 +0.1/50分） |
| difficulty_level | int | 当前难度等级编号，初始 0，由 score // SCORE_THRESHOLD_INTERVAL 计算得出 |

**关系**：
- 与 Game 实例生命周期绑定，由 Game.reset() 初始化/清零
- difficulty_level 是 difficulty_multiplier 的驱动因子：multiplier = level * DIFFICULTY_INCREMENT

### 8.2 DifficultyConfig（config.py 常量）

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| SCORE_THRESHOLD_INTERVAL | int | 50 | 每多少分提升一级难度 |
| DIFFICULTY_INCREMENT | float | 0.1 | 每级难度增加的倍率值 |
| MAX_DIFFICULTY_MULTIPLIER | float | 4.0 | 难度倍率上限，由 BASE_TICK_INTERVAL / MIN_TICK_INTERVAL - 1 推导 |

**关系**：
- 被 Game._get_current_tick_interval() 引用以计算实际 tick 间隔
- MAX_DIFFICULTY_MULTIPLIER 与 BOOST_MULTIPLIER 叠加后保证 tick >= MIN_TICK_INTERVAL

## 9. 边界情况

| 场景 | 预期行为 |
|------|----------|
| 分数恰好跨越阈值边界（如从 49 到 51 一次吃到 2 分食物） | 难度应正确递增 1 级而非跳级（按最终分数 / 阈值计算 level 即可） |
| 极端高分（假设分数 9999，difficulty_level = 199，multiplier = 19.9） | 远超 MAX_DIFFICULTY_MULTIPLIER，钳位至上限 4.0 对应的 tick 间隔 |
| 手动 boost 激活期间跨越难度阈值 | 两个 multiplier 独立变化，叠加后 tick 间隔平滑变化，不会因为 boost 而跳过难度提升 |
| boost 释放瞬间（从高倍率回到仅 difficulty） | tick 间隔不会跌破下限，变化幅度由 boost_multiplier 决定，可感知但不突兀 |
| 连续快速吃食物（短时间内多次跨越阈值） | 每次 tick 计算独立，难度逐级递增不会丢失中间等级 |
| reset() 后立即开始移动 | difficulty_multiplier 已清零，tick 间隔为 100ms，与游戏首次启动行为一致 |
| 窗口失焦/暂停期间 | tick 不推进，分数不变化，难度不变，恢复后从当前难度继续 |
| diff_multiplier 浮点累加精度（0.1 累加 100 次） | 使用 level * increment 计算而非逐次累加，避免累积误差（0.1*100 = 10.0 精确） |

## 10. 集成点

| 集成目标 | 文件 | 说明 |
|----------|------|------|
| Game._get_current_tick_interval() | game.py | 现有方法返回基础 tick 间隔，需扩展为 BASE / (1 + difficulty_multiplier + boost_multiplier)，并施加 max(result, MIN_TICK_INTERVAL) 保护 |
| Game._update_boost_state() | game.py | 现有 boost 状态更新逻辑，保持不变；boost_multiplier 照常产出，由 _get_current_tick_interval() 统一叠加 |
| config.py | config.py | 新增 SCORE_THRESHOLD_INTERVAL、DIFFICULTY_INCREMENT、MAX_DIFFICULTY_MULTIPLIER 三个模块级常量 |
| Renderer.draw_frame() 或等价 HUD 渲染入口 | renderer.py | 扩展参数以接收 difficulty_level 和综合倍率，新增 HUD 行渲染 |
| Game.reset() | game.py | 扩展重置逻辑以清零 difficulty_multiplier 和 difficulty_level |
| Game.update() 或主循环中 score 变更处 | game.py | 在分数更新后同步重新计算 difficulty_level 和 difficulty_multiplier |

## 11. 约束

- BASE_TICK_INTERVAL = 100ms（现有基准，不可变更）
- MIN_TICK_INTERVAL = 20ms（现有下限，作为速度上限硬约束）
- 难度倍率与 boost 倍率采用加法叠加（1 + diff + boost），不使用乘法
- 难度递增为台阶式离散变化，非连续渐变（每跨越一个阈值触发一次倍率变更）
- 仅修改 game.py / config.py / snake.py / renderer.py 中相关内容，不新增文件
- 不引入新的第三方依赖，不改变现有 Pygame 版本要求
- 难度倍率计算逻辑必须置于 _get_current_tick_interval() 调用链中

## 12. 假设条件

- 难度仅通过蛇的移动速度体现，不涉及其他机制变化
- 难度倍率的递增方式为：达到每个分数阈值后，难度倍率累加固定增量（例如每 50 分 +0.1x 基础速度）
- 难度倍率上限以现有 MIN_TICK_INTERVAL（20ms）为硬约束，即 BASE_TICK_INTERVAL / (boost_multiplier + difficulty_multiplier) 不低于 20ms
- 难度倍率与手动 boost 倍率为加法叠加（diff_multiplier + boost_multiplier），而非乘法叠加 —— 加法更可控、不易越过 tick 下限
- HUD 中新增一行显示难度等级（基于当前分数区间计算）和当前综合倍率
- 假设现有渲染帧率 60FPS 不变，难度递增不要求刷新率变化

## 13. 外部依赖

| 依赖项 | 说明 |
|--------|------|
| 现有 `_get_current_tick_interval()` 方法（game.py） | 难度倍率需作为新的除数因子介入 |
| 现有 `boost_multiplier` 计算链路（Snake.boost_state + Game._update_boost_state） | 需要明确两套倍率的叠加方式 |
| 现有 config.py 配置体系 | 需新增难度相关常量（分数阈值、每级倍率增量、最大难度倍率上限等） |
| 现有 Renderer.draw_frame() 的 HUD 渲染 | 需扩展以接收并绘制难度等级信息 |
| 现有 Game.reset() | 需在重置链路中覆盖难度倍率复位 |
