- **整理者 P0#2（assert 替换为 if-raise）**：假设 #11 和 D11 已显式接受 `python -O` 下 assert 失效的风险，并给出风险评估（"最多异常移动一步后 GAME_OVER——不会导致崩溃、数据损坏或安全漏洞"）。Snake Game 从不以 `-O` 运行，且 direction 唯一写入路径（consume_direction 返回值来自 DIRECTION_MAP）加上入队校验链已构成多层防御。此为已接受的已知风险，无需新行动。
- **整理者 P0#3（enqueue 反向检查覆盖 committed_direction）**：该场景在消费阶段已被 D02 的 while 循环兜底——若 LEFT 在消费时恰与当前 committed_direction 相反，while 循环会跳过它。若 LEFT 在消费时非反向，则执行 LEFT 是 FIFO 队列的正确语义（用户按了 LEFT 就应执行 LEFT）。增加 enqueue 阶段的 committed_direction 检查反而会误杀合法序列。不达 P0，降为 P2 文档化边界说明。

#### 1. FR-006 焦点丢失边界说明：文档化"不清空队列"的设计权衡

**来源**: 批评家（原始 P0，校准为 P2）
**涉及**: 需求分析 FR-006 验收标准 AC-6.3 + 边界情况表 #13

**问题**: 当前设计（焦点丢失不清空队列，仅在 reset 时清空）面临一个 UX 边界：若玩家暂停/失焦期间误触方向键，恢复 RUNNING 后队列中可能包含非玩家本意的方向。批评家主张清空队列以防意外转向致死。但反方向权衡同等成立：若玩家在暂停前已通过快速按键构建了有效队列（如连续 3 个方向的 L 形转弯序列），焦点丢失自动清空会静默丢弃玩家的合法输入，破坏输入透明度承诺（NFR-U01）。

**分析**: 本地桌面游戏的焦点丢失远不如 Web 应用频繁（触发场景主要是 Alt+Tab 或系统通知）。FR-006 边界情况 #13 已规定"不消费方向"，误触方向在暂停期间仅入队不执行。最坏情况：恢复后最多执行队列中残留的误触方向，导致一次意外转后触发碰撞 GAME_OVER —— 按下 R 即可重置。权衡的天平倾向"不清空"：保护合法输入比防御罕见误触更有价值。

**建议**: 无需修改行为。在 FR-006 或边界情况 #13 处补充一句权衡说明：
> "设计选择不清空队列以保护焦点丢失前的合法输入序列。焦点丢失期间误触导致队列污染的风险存在于理论层面（Alt+Tab 时物理按键误触），在本地桌面游戏中实际发生概率极低；最坏情况下残留方向至多导致一次意外 GAME_OVER，按 R 重置即可恢复。若将来迁移至移动端或多点触控场景，需重新评估。"

**不修后果**: 无功能影响。仅当未来有人质疑"为什么不清空"时缺乏可追溯的设计依据

#### 2. 概要设计补充 enqueue_direction 队非空时反向过滤仅检查队尾的边界说明

**来源**: 批评家（原始 P0，校准为 P2）
**涉及**: 概要设计 FR-002 反向过滤逻辑 + D02 消费阶段 while 循环

**问题**: enqueue_direction 在队列非空时，反向检查仅与 `queue[-1]`（队尾）比较。批评家指出：若队列为 `[RIGHT, UP]` 且玩家按 LEFT，LEFT 非 UP 的反向而入队，后续 RIGHT 被消费后 committed_direction 变化，LEFT 可能在消费时被意外执行（而非作为 RIGHT 的反向被拦截）。

**分析**: 此场景下 LEFT 被消费时，committed_direction 已因 RIGHT 的消费而变为 RIGHT，LEFT 恰是 RIGHT 的相反方向——D02 的 while 循环会在消费时检测到这一反向关系并跳过 LEFT。而若 LEFT 在消费时恰好非反向（如前序条目已使蛇转向其他方向），则执行 LEFT 是 FIFO 队列的正确语义——玩家按了 LEFT，就应按序执行。enqueue 阶段增加 committed_direction 检查反而会误杀合法序列（如队列 `[RIGHT, UP]`，玩家按 LEFT 意图在 RIGHT 后左转——这在 enqueue 阶段被 committed_dir 检查拦截是错误的）。

**建议**: 无需修改入队逻辑。在 D02 或 FR-002 处补充一句边界说明：
> "enqueue_direction 反向过滤仅检查 `queue[-1]`（连续反向）。队列中非连续的反向条目（如 `[RIGHT, UP]` 中 RIGHT 与后续可能的 LEFT）在入队阶段不被拦截，但 D02 消费阶段 while 循环会基于消费时的 committed_direction 重新检测并跳过反向条目。此设计在入队效率（O(1) 无遍历）与消费安全性之间取得平衡。"

**不修后果**: 无功能影响。消费阶段 while 循环已提供安全网

#### 3. process_events 方法 docstring 标注队列副作用

**来源**: 批评家（原始 P1，校准为 P2）
**涉及**: 概要设计 InputHandler 组件接口表 + 需求分析 集成点表

**问题**: process_events 在本次设计中会直接调用 enqueue_direction 写入队列（有状态副作用），但接口仍返回命令字典（`{'action': str, 'boost': bool}`）。批评家指出：若单元测试仅 mock process_events 返回值而不触发实际入队，将漏掉队列状态变更，形成覆盖盲区。

**分析**: process_events 写入队列是其在此设计中的核心职责，不应称为"副作用"。但批评家对接口语义不对称的观察是准确的：返回 dict 暗示方法是无副作用的查询，实际上方向键处理分支触发了队列变更。在 ~1400 行项目中重构为 enqueue-only 无返回值方法属于过度工程（需修改 Game._handle_command 的事件分发流程），但 docstring 标注成本为零。

**建议**: 在 InputHandler 组件接口表中的 process_events 条目，或实施阶段 process_events 方法的 docstring 中，显式标注：
> "本方法在处理方向键 KEYDOWN 事件时会调用 enqueue_direction() 修改内部队列状态。返回值中的命令字典（action/boost）不包含队列变更信息——集成测试验证队列行为时需通过 get_queue_snapshot() 或 consume_direction() 返回值进行断言，而非依赖 process_events 返回字典。"

**不修后果**: 实施者可能仅基于 process_events 返回 dict 编写 mock 测试，漏掉队列入队行为验证

#### 4. 以 `if __debug__: print()` 替代 `logging` 模块

**来源**: 实业家（原始 P2，维持 P2）
**涉及**: 概要设计 3.技术栈 + D08 设计决策

**问题**: 当前设计使用 Python `logging` 模块输出入队丢弃和消费跳过的 DEBUG 日志。"logging 模块"引入了 Logger 实例化、级别配置、格式化等框架概念，在 ~1400 行本地游戏中属于基础设施膨胀。等效方案：`if __debug__: print(f"[INPUT_QUEUE] ...")` 在标准 Python 模式下提供同等可观察性，`python -O` 运行时自动消除（与 D11 的 assert 策略一致），零配置、零新增 import。

**分析**: logging 和 `__debug__` + print 均为 Python 标准库，不涉及依赖变更。但 logging 的配置心智负担（basicConfig、getLogger、级别继承）在小型项目中是不成比例的开销。设计决策 D08 的核心目标（开发阶段可观察性、生产零开销）二者均可满足。`if __debug__` 方案与 D11（assert 策略）的技术哲学一致。

**建议**: 在概要设计 3.技术栈和 D08 中将 `logging` 改为 `if __debug__: print(...)`。标注 `python -O` 下自动消除，无运行时开销。若将来迁移至后台服务或多模块日志聚合场景，可重新评估 logging 模块。

**不修后果**: 无功能影响。当前选型（logging）也能正常工作，仅稍显笨重
