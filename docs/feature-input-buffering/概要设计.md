# 概要设计

> **功能**: 方向键输入缓冲系统 (input-buffering) | **阶段**: 概要设计 (P2) | **角色**: 软件架构师 | **日期**: 2026-07-11

## 1. 架构模式

### 选择

**生产者-消费者 + 有界 FIFO 缓冲队列**（嵌入现有 Game Loop + Command-Mediated 架构）

### 理由

当前逐帧单槽缓冲（每渲染帧仅保留最后一个有效方向键，立即调用 `change_direction()` 覆盖蛇方向）的根因在于：方向输入的生产速率（渲染帧 ~60Hz）远高于消费速率（逻辑 tick ~10-50Hz），中间按键被后续帧覆盖丢失。

引入有界 FIFO 缓冲队列（`collections.deque`, maxlen=4），将渲染帧作为生产者（`process_events` 中方向键直接入队），逻辑 tick 作为消费者（`_update` 中 `consume_direction` 出队写入 `snake.direction`），精准解耦生产与消费速率。现有 `committed_direction` 机制、accumulator 速率控制和模块化组件划分天然适配此架构改造——改动集中在 InputHandler、Game、Snake、Renderer 四个模块，同时断开 `game -> input_handler -> game` 循环依赖。

对比备选方案（参见第 6 节设计决策 D-01），此方案改动面最小、符合现有架构范式、不引入额外线程或全局状态。

## 4. 数据流

```
每渲染帧 (RENDER_FPS=60):

renderer.tick()
  -> dt (ms)

InputHandler.process_events(committed_dir, state)
  -> 遍历 Pygame 事件队列
  -> 方向键 KEYDOWN: 直接调用 enqueue_direction(dir, committed_dir)
     |- 有效性校验 (DIRECTION_MAP)
     |- 队尾同向去重 (与 _direction_queue[-1] 比较)
     |- 反向过滤 (队空时以 committed_dir 为参照，否则以队尾为参照)
     |- 追加到 _direction_queue (deque, maxlen=4)
     |- 丢弃时输出 DEBUG log
  -> 返回命令字典 (无 direction 键: quit/restart/pause/resume/none)

Game._handle_command(command)
  -> 仅处理 quit / restart (direction 分支已移除)

accumulator += dt
若 accumulator >= tick_interval 且 state == RUNNING 且未暂停:
  accumulator -= tick_interval
  Game._update():
    -> 新增: direction = input_handler.consume_direction(snake.committed_direction)
       |- while 循环 popleft() + 反向检测 (与 committed_dir 比较)
       |- 跳过反向条目 (DEBUG log)
       |- 至多返回 1 个有效方向或 None
    -> 若 direction 非 None: snake.direction = direction
    -> snake.move_and_grow(grow_flag):
       |- assert self.direction in self.VALID_DIRECTIONS (防御，python -O 零开销)
       |- committed_direction = self.direction
    -> 碰撞检测

renderer.draw_frame(..., direction_queue=input_handler.get_queue_snapshot())
  -> renderer.draw_hud(..., direction_queue=list_of_directions)
     |- 非空时在 HUD 区域渲染方向箭头序列 (^ v < >)

仅 Game.reset() 时:
  input_handler.flush_queue()  -> queue.clear()
```

## 6. 设计决策

| 决策 | 理由 | 备选方案 | 权衡 |
|------|------|----------|------|
| **D-01: 队列存储在 InputHandler 而非 Snake** | InputHandler 管理所有输入状态（暂停/加速/队列）职责内聚；Snake 是纯实体不应持有事件衍生数据结构。`process_events` 中方向键直接入队不需跨层传参 | Snake 持有队列：入队更方便（不需传 committed_dir），但破坏 Snake 纯实体性质，且需 Snake 感知 InputHandler 框架概念 | 牺牲入队便利（需传 committed_dir 参数）换取清晰职责边界和模块内聚 |
| **D-02: 消费阶段 while 循环跳过连续反向条目** | 入队反向过滤基于当时参照方向（队尾或 committed_dir），但消费时 committed_direction 可能因前序 tick 消费而改变。while 循环防止队列被已变无效的反向条目永久阻塞（例如：队列 [LEFT, RIGHT]，tick1 消费 LEFT 后蛇方向改变，tick2 消费 RIGHT 时 RIGHT 与新的当前方向相反） | 只检查一个方向：若该方向在消费时已变反向，返回 None 但保留在队首，下一 tick 继续检查——导致队列永久阻塞，后续有效方向永不出队 | 每 tick 至多 O(4) 内部循环（远低于 100us 阈值），换取队列自洁能力和不阻塞保证。时间复杂度 O(1) 均摊 |
| **D-03: 方向键在 process_events 中直接入队，不经过 command dict** | 方向键生命期完全在 InputHandler 内部闭环，消除 command dict 的序列化/反序列化中间层；`_handle_command` 职责更清晰（仅 quit/restart 两个 action） | 保留 direction action 改为调用 `enqueue_direction`：增加不必要的间接层和 command dict 中 direction 键的序列化开销 | `process_events` 内部变复杂（方向键处理从"设定 last_direction"改为"调用 enqueue_direction"），但消除 direction 命令的跨层传递和约 10 条 `command['direction']` 集成测试迁移成本 |
| **D-04: 删除 Snake.change_direction()（含测试迁移策略，详见 6.1 节）** | 方向变更唯一入口变为队列消费 -> 写入 direction；保留空壳或改为 setter 会误导维护者。删除后方向变更路径唯一化，降低认知负担 | (1) 保留为内部 setter：引入 `Snake -> InputHandler` 反向依赖；(2) 保留为空壳：误导维护者 | ~19 条单元测试迁移成本 + ~10 条集成测试适配成本，但彻底消除歧义和潜在依赖，方向变更流线单一可追溯 |
| **D-05: 暂停/GameOver/Victory 时不清空队列，仅 reset 时清空** | 静默丢弃不符合玩家预期——玩家在非 RUNNING 状态下的输入可能是操作过快的遗留（如最后一刻按下转向却未执行）。保留至下次 reset 代价极小（队列最多 4 条目，不消费），符合最小惊讶原则 | 所有状态变更清空：静默丢弃玩家输入，可能引发"按了键但没反应"的困惑 | GameOver 后直接退出不 reset 时残留方向随实例销毁（无副作用）；若 reset 则 flush_queue 统一清空。唯一小风险：暂停期间大量方向键入队覆盖旧条目（但暂停通常短暂） |
| **D-06: Snake.move_and_grow() 开头新增 assert 防御（P0）** | 删除 `change_direction()` 后 direction 唯一写入路径为 `consume_direction`（经 `Game._update`）；若未来维护引入逻辑缺陷导致非法值（如 (2,2)）漏过，assert 可立即在开发阶段捕获而非静默产生异常移动 | (1) 在 `consume_direction` 调用处加校验：仅保护一个入口，若 `direction` 被其他代码直写则漏过；(2) `move_and_grow` 中 if-raise：运行时始终有开销 | 运行时开销为零（`python -O` 移除），防御非法值污染物理步进；标准 Python 模式下可立即定位逻辑缺陷 |
| **D-07: 每逻辑 tick 开销不超过 100us（NFR-P02 量化指标）** | 原 1us 在 CPython 物理不可达（deque.popleft + 元组比较 + 函数调用已超 ~50ns）；100us 在 20ms tick（极端难度下限）下占比 0.5%，在 100ms tick（基准）下占比 0.1%，实际最坏情况（while x4）约 <50us | 定性表述"不对帧率产生可观测影响"：不可量化验证，性能回归测试无基准 | 量化指标供性能回归测试，同时物理可达（`time.perf_counter` 可测） |
| **D-08: Debug 可观察性——入队/消费丢弃时输出 DEBUG log** | 入队丢弃（同向/反向/FIFO 溢出）和消费跳过反向时输出 DEBUG 日志；生产环境默认 WARNING 级别静默；开发者开启 `logging.DEBUG` 可追溯队列行为，辅助问题诊断和测试验证 | 不使用日志：队列行为无追溯手段，测试和调试依赖断点/HUD 肉眼观察 | 4 条日志位于热路径但为 DEBUG 级别，生产零开销；开发阶段提供关键可观察性 |
| **D-09: 固定队列容量 maxlen=4，不提供可配置项** | 基于人体工程学：正常人手速极限约 10-15 次/秒，在 20ms tick（最极端难度）下 4 tick 预输入窗口即 80ms，足以覆盖 1-2 次按键；更多容量增加认知复杂度且不符合需求范围（宏/脚本场景排除） | 可配置队列长度：增加配置复杂度，测试组合爆炸，当前无实际需求驱动 | 固定值简化设计和测试，4 是经计算的合理上限；未来如需可改为 `config.QUEUE_MAXLEN` |
| **D-10: 队空时以 committed_direction（而非 direction）为反向过滤参照** | `direction` 可能已被前序 `enqueue_direction` 修改但尚未经 `move_and_grow` 消费（如帧 1 LEFT 入队写入 direction，帧 2 入队以 direction=LEFT 为参照则错误放行 DOWN）。`committed_direction` 始终对应上一次物理移动方向，是最可靠的防自杀基准 | 以 `direction` 为参照：引入跨帧反向检测的时间窗口漏洞 | 无权衡——此选择消除跨帧反向绕过漏洞，正确性优先 |
| **D-11: assert 在 python -O 下失效是已接受的风险** | Python `-O` 模式移除所有 assert 语句。`Snake.move_and_grow()` 内的 `assert self.direction in self.VALID_DIRECTIONS` 仅用于开发阶段及早捕获非法方向值；正常游戏运行使用标准 Python，assert 生效。`python -O` 通常用于性能敏感的生产部署——此游戏为本地桌面应用，标准模式是常规用法。若 `-O` 下非法 direction 值漏过，后果是蛇朝非标准方向移动一步（可能出界或产生异常帧），下一 tick 通常会被边界碰撞捕获为 GAME_OVER——不会导致崩溃、数据损坏或安全漏洞 | 替换为 `if direction not in VALID_DIRECTIONS: raise ValueError`：运行时始终有开销，且此类逻辑缺陷在开发阶段应已被 assert 捕获 | 标准模式下零额外开销的防御性校验；`-O` 下的风险窗口极小且故障模式有限（最多异常移动一步后 GAME_OVER） |
| **D-12: HUD 队列箭头渲染位置** | 在现有 HUD 信息区域（800x40 像素）内，分数（左侧）和难度信息（居中）之间新增队列显示。箭头使用 `font_small` 字体、`COLORS.TEXT` 颜色，单空格分隔。队列为空时该区域不渲染额外内容。箭头符号映射：`(0,-1)` -> `^`，`(0,1)` -> `v`，`(-1,0)` -> `<`，`(1,0)` -> `>` | (1) 队列显示在游戏区域左下角；(2) 独立第二行 HUD | 选择居中偏左位置兼顾可读性和不干扰现有布局。P2 优先级意味着实现上保持简单——不引入新的布局系统或独立 HUD 行 |

### 6.1 测试迁移策略

本节详述 D-04（删除 `Snake.change_direction()`）和 D-03（`process_events` 返回格式变更）所需的测试迁移工作。

#### 6.1.1 Snake.change_direction() 单测迁移（~19 条）

当前 `test_snake.py` 中 `TestChangeDirection` 类包含约 10 条测试，`TestSnakeCommittedDirection` 类包含约 9 条（总计 ~19 条）依赖 `change_direction()` 方法。这些用例覆盖方向变更、反向拦截、无效方向拒绝、同向接受、跨帧反向拦截等行为。

**迁移策略**：将 `change_direction()` 的测试语义映射到 InputHandler 层：
