## 1. 复杂度评估

### 1.1 模块难度评分

| 模块 | 难度(1-5) | 理由 |
|------|-----------|------|
| config.py | 1 | 仅新增约10个常量 + 一个模块级校验函数，纯数据定义，无耦合逻辑。 |
| snake.py | 2 | 新增 boost_state 字典属性 + 两个 property + reset() 扩展，改动量约15行。逻辑简单，但需确保 reset() 在基类调用链中不被遗漏。 |
| input_handler.py | 3 | 引入 pygame.key.get_pressed() 轮询创建双重输入路径（KEYDOWN事件+持续轮询），新增 _boost_active 标志位和命令字典 boost 字段，焦点丢失需同步清除两种输入状态。事件模型与轮询模型的协调是中等复杂度。 |
| game.py | 5 | 主循环从 clock.tick(10)+每帧逻辑 重构为 渲染60fps+时间累加器+可变tick间隔 的解耦架构。需实现平滑过渡线性插值、tick间隔 clamp 防护、MAX_CATCHUP_STEPS 螺旋死亡保护、状态转换双重点复位、非游玩态加速禁用。这是整个功能最核心且风险最高的变更。 |
| renderer.py | 2 | 4个方法签名扩展 is_boosting 参数，蛇身/蛇头颜色分支选择，HUD条件渲染BOOST文字，tick()改用RENDER_FPS。改动分散但每处修改简单明确。需注意_freetype字体与新增文字渲染的兼容性。 |
| main.py | 1 | 无接口变更，焦点丢失boost复位由InputHandler内部协作完成。仅需验证事件分发顺序是否符合预期。 |
| 测试文件 (7个文件, 373个用例) | 4 | game.py 主循环重构使所有依赖固定FPS行为(tick-per-frame)的测试失效；renderer方法签名变更需更新全部调用侧测试；input_handler新增boost字段需新增覆盖用例(按键按住/释放/焦点丢失/非游玩态)。测试回归工作量接近甚至超过功能实现本身。 |

### 1.2 整体复杂度

**high**

整体复杂度评定为 high，原因有三：
1. game.py 主循环从单一时钟驱动重构为渲染/逻辑解耦架构，属于根本性变更，引入时间累加器、可变tick间隔、螺旋死亡保护等新机制
2. input_handler 需要从纯事件驱动模型扩展到事件+轮询双路径，两个输入通路的状态一致性是关键挑战
3. 373个现有测试用例因主循环重构和方法签名变更面临大规模回归更新，测试工作量可能超过功能实现

## 2. 风险评估

| 类型 | 严重度 | 描述 | 缓解措施 |
|------|--------|------|----------|
| 技术 | **严重** | 主循环解耦重构: 从 clock.tick(10) 驱动逻辑+渲染的统一循环改为 60fps渲染 + 时间累加器可变tick。若累加器逻辑有 bug（如 dt 精度丢失、tick_interval 边界条件处理不当），可能导致蛇步进不均匀、卡顿或螺旋死亡假死。 | 严格限制 MAX_CATCHUP_STEPS=5；tick_interval 计算后 clamp 到 [20ms, BASE_TICK_INTERVAL]；单元测试覆盖累加器边界（dt=0, dt 极大, tick_interval=20ms 极限）；beta 测试覆盖 5x 加速连续运行 60 秒。 |
| 技术 | **高** | 输入双路径一致性: KEYDOWN 事件处理和 get_pressed() 轮询各自维护按键状态。OS 键盘重复事件(KEYDOWN 重复投递)可能与轮询状态不同步；焦点丢失后 get_pressed() 状态更新存在 Pygame 已知的平台差异。 | Boost 激活仅以 get_pressed() 轮询为唯一真实来源（KEYDOWN/KEYUP 不直接控制 boost_state，仅作辅助日志）；焦点丢失事件中强制清除 _boost_active；集成测试覆盖 Windows/Linux/macOS 三平台焦点切换行为。 |
| 技术 | **中** | 测试大规模回归: 373个测试用例中 game.py 相关测试(55个)因循环重构全部需要审查更新；renderer 方法签名变更影响所有调用侧测试。测试更新工作量可能占整体工时的30-40%。 | 先更新核心模块测试(game/input/renderer)，再更新集成测试；保持测试粒度与现有文件结构一致；为新功能新增 30+ 专用用例。 |
| 性能 | **中** | 5x加速时tick间隔仅20ms，渲染仍为60fps(~16.6ms)，逻辑tick可能短于帧间隔。累加器需正确处理逻辑快于渲染的情况，避免丢帧或 tick 堆积。 | MAX_CATCHUP_STEPS=5 限制单帧最多执行5次逻辑更新；若 accumulator 持续超过 5*tick_interval 则丢弃多余时间防止螺旋死亡；性能 profile 验证 5x 加速下帧率 >=55fps。 |
| 依赖 | **低** | pygame.key.get_pressed() 在部分平台(Linux Wayland/某些输入法环境下)可能返回过时的键状态，导致按住加速键无法被检测。 | config.py 中 BOOST_KEY 可配置降级方案；若连续多帧未检测到按键但 KEYUP 事件也从未触发，记录 WARNING 并自动降级不加速；文档注明已知平台限制。 |
| 安全 | **低** | 配置参数注入: 若用户通过非常规方式修改 config.py 注入极端值(如 BOOST_SPEED_MULTIPLIER=-1 或 1000)，tick_interval 计算可能产生负值或除零。 | 模块加载时 _validate_boost_config() 对所有加速参数进行边界校验和 clamp；运行时 _get_current_tick_interval() 输出再次 clamp>=20ms 作为兜底。双重防护确保即使绕过加载时校验也不会 crash。 |

## 9. 附录：建议实施方案详述

分3阶段迭代实施，每阶段可独立验证：

**阶段1 (2.5天) - 基础设施**: config.py 新增常量与校验 -> snake.py boost_state 属性 -> renderer.py 方法签名扩展（先传 False 保持行为不变）-> input_handler.py get_pressed 轮询 + boost 字段。此阶段结束时 game.py 仍为原样，加速输入可检测但尚未生效。

**阶段2 (3天) - 核心重构**: game.py 主循环解耦（时间累加器+可变tick）+ 加速状态管理 + 速度平滑过渡 + 状态复位 + 非游玩态禁用。此阶段是风险最高部分，完成后需充分集成测试。

**阶段3 (1.5天) - 测试与打磨**: 更新全部回归测试 + 新增加速专用测试用例(30+) + 边界情况验证（5x长按60秒/快速连打/焦点切换）+ 性能profile确认帧率要求。

不建议一次性全部实施，每阶段完成后 review 并验证，降低回滚成本。
