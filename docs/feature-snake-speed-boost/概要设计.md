# 概要设计

> **功能**: 蛇移动速度加速 (Snake Speed Boost on Key Hold) | **阶段**: 概要设计 (P2) | **角色**: 软件架构师 | **日期**: 2026-07-08

## 1. 架构模式

### 1.1 模式选择

**分层架构 (Layered Architecture) + 解耦游戏循环 (Decoupled Game Loop)**

当前项目已采用分层架构：`main.py`（入口/编排） -> `game.py`（协调层） -> `snake.py`/`food.py`（领域实体）+ `renderer.py`（渲染层）+ `input_handler.py`（输入层）。加速功能在各层增量扩展，不引入新架构模式。

### 1.2 理由

- **分层架构**: 变更局限在现有模块内，每一层职责明确（输入检测/状态管理/视觉渲染），改动影响面可控
- **解耦游戏循环**: 现有架构中渲染帧率与游戏逻辑 tick 强绑定（FPS=10 同时控制渲染和逻辑更新），加速功能需要蛇移动频率独立于渲染帧率。引入时间累加器将逻辑 tick 与渲染帧率解耦：渲染固定在可配置帧率（如 60fps）以保证视觉平滑，游戏逻辑按可变 tick 间隔独立推进
- 解耦后的游戏循环是本次变更最大的架构调整，但它是实现平滑速度过渡（FR-007）和保持视觉体验一致的必要前提

#### 2.2.2 input_handler.py (输入处理模块)

| 维度 | 内容 |
|------|------|
| **职责** | 检测加速键按下/释放状态（持续轮询），产出 boost 命令字段；保持现有事件驱动方向处理不变 |
| **新增接口** | `is_boost_pressed() -> bool` 公开方法，每帧由 Game 调用 |
| **现有接口变更** | `process_events()` 返回的命令字典新增可选 `boost` 键（bool）；焦点丢失时自动复位 `_boost_active` 内部标志 |
| **依赖** | `config.BOOST_KEY`, `config.GameState`, `pygame.key.get_pressed()` |

关键设计：加速键使用**轮询检测**（`pygame.key.get_pressed()`）而非事件驱动（`KEYDOWN`/`KEYUP`），因为：
- 加速是持续状态（hold-to-activate），不是瞬时事件
- 轮询天然支持"按住不放"的语义，无需区分 key-repeat 和 hold
- 方向键保持原有事件驱动，两者在 InputHandler 内共存——每帧既消费事件队列（方向/R/Q），也轮询加速键状态

命令字典结构变更：

```python
# 现有
{'action': 'direction', 'direction': (0, -1)}

# 扩展后
{'action': 'direction', 'direction': (0, -1), 'boost': True}
```

焦点丢失时 `_boost_active` 内部标志立即复位，再次调用 `is_boost_pressed()` 只在明确收到 `KEYDOWN` 后才返回 True（通过标志位和轮询组合）。

#### 2.2.4 game.py (游戏协调模块)

| 维度 | 内容 |
|------|------|
| **职责** | 实现解耦游戏循环（时间累加器）；管理 boost 状态生命周期（激活、平滑过渡、复位）；在状态转换时触发加速复位；非 RUNNING 状态下忽略加速 |
| **新增接口** | 内部方法: `_update_boost_state(dt, boost_pressed)`, `_get_current_tick_interval()` |
| **现有接口变更** | `run()` 循环重构为解耦模式；`reset()` 追加 boost 复位 |
| **依赖** | `config` 所有 boost 常量, `snake.Snake`, `input_handler.InputHandler`, `renderer.Renderer` |

核心变更——`run()` 方法重构为解耦游戏循环：

```python
def run(self) -> None:
    running = True
    logic_accumulator: float = 0.0  # 逻辑累积时间 (ms)

    while running:
        dt = self.renderer.tick()  # clock.tick(RENDER_FPS) -> ms

        # 1) 输入处理（每渲染帧执行）
        command = self.input_handler.process_events(...)
        running = self._handle_command(command)

        # 2) 更新 boost 状态（每渲染帧执行，用于平滑过渡）
        boost_pressed = self.input_handler.is_boost_pressed()
        self._update_boost_state(dt, boost_pressed)

        # 3) 游戏逻辑（按 tick 间隔执行，可每帧执行 0 次、1 次或多次）
        tick_interval = self._get_current_tick_interval()
        logic_accumulator += dt
        while logic_accumulator >= tick_interval:
            self._update()  # 蛇移动一步 + 碰撞检测 + 食物逻辑
            logic_accumulator -= tick_interval
            # 防止 spiral-of-death: 限制每帧最多 catch-up 步数
            if tick_interval <= 0:
                break

        # 4) 渲染
        self.renderer.draw_frame(self.snake, self.food, self.score,
                                  self.state, self.snake.is_boosting)
        pygame.display.flip()
```

**平滑过渡实现 (`_update_boost_state`)**: 使用线性插值在约 150ms 内更新 `snake.boost_state['current_multiplier']`。按下加速键时 target=BOOST_SPEED_MULTIPLIER，释放时 target=1.0，每帧按 `dt / BOOST_TRANSITION_SECONDS` 比例逼近目标值。

**参数校验**: `BOOST_SPEED_MULTIPLIER` 在 Game 初始化时校验，超出 [1.0, MAX_BOOST_MULTIPLIER] 范围时 clamp 并记录警告（NFR-S001）；tick_interval 计算结果校验，不允许 <= 0，异常时回退到 BASE_TICK_INTERVAL（NFR-S002）。

## 4. 数据流

### 4.1 主数据流（每帧）

```
键盘硬件
  │
  ├─► pygame.event.get() ──► InputHandler.process_events()
  │                              │
  │                              ├─► command {action, direction}     ──► Game._handle_command()
  │                              └─► 内部更新 _boost_pressed 标志
  │
  └─► pygame.key.get_pressed() ──► InputHandler.is_boost_pressed()
                                       │
                                       v
                                  Game._update_boost_state(dt, boost_pressed)
                                       │
                                       ├─► 计算 target_multiplier (1.0 或 BOOST_SPEED_MULTIPLIER)
                                       ├─► 线性插值更新 snake._boost_state['current_multiplier']
                                       └─► snake._boost_state['is_active'] = (current_multiplier > 1.0)
                                              │
                                              v
                                  Game._get_current_tick_interval()
                                       │
                                       v
                                  tick_interval = BASE_TICK_INTERVAL / snake.boost_multiplier
                                       │
                                       v
                                  Game._update() (条件执行: accumulator >= tick_interval)
                                       │
                                       v
                                  Renderer.draw_frame(..., is_boosting=snake.is_boosting)
                                       │
                                       ├─► draw_snake() 选择 boost 颜色
                                       └─► draw_hud() 渲染 "BOOST" 文字
```

### 4.2 状态转换数据流

```
事件触发                              Boost 状态变化
─────────────────────────────────────────────────────────
Game._update() → state=GAME_OVER     snake.boost_state 全部复位 -> 1.0
InputHandler: R key → reset()        snake.boost_state 全部复位 -> 1.0
InputHandler: P key (未来)           snake.boost_state 全部复位 -> 1.0
InputHandler: WINDOWFOCUSLOST        _boost_pressed=False, boost_state 复位
加速键释放 (is_boost_pressed==False)  target_multiplier → 1.0 (平滑过渡)
加速键按下 (is_boost_pressed==True)   target_multiplier → BOOST_SPEED_MULTIPLIER (平滑过渡)
GameState != RUNNING                 is_boost_pressed() 被忽略，boost 不激活
```

### 4.3 平滑过渡数据流（FR-007）

```
加速键按下瞬间:
  current_multiplier = 1.0
  target_multiplier = BOOST_SPEED_MULTIPLIER (e.g., 2.0)
  每帧: current_multiplier += (target - current) * (dt / BOOST_TRANSITION_SECONDS)
  当 |current - target| < 0.01 时，直接设置为 target

加速键释放瞬间:
  target_multiplier = 1.0
  每帧: current_multiplier += (target - current) * (dt / BOOST_TRANSITION_SECONDS)
  当 |current - target| < 0.01 时，直接设置为 target
```

`current_multiplier` 在 1.0 到 BOOST_SPEED_MULTIPLIER 之间连续变化，tick_interval 随之连续变化，实现平滑的速度过渡。过渡完成后（差距 < 0.01），直接 snap 到目标值避免无限逼近。

## 5. 集成点

| 编号 | 名称 | 类型 | 描述 |
|------|------|------|------|
| IP-001 | config.py | internal | 加速配置常量源。Game 和 Renderer 在初始化时读取。加载时执行参数校验：`BOOST_SPEED_MULTIPLIER` clamp 到 `[1.0, MAX_BOOST_MULTIPLIER]`，`BOOST_KEY` 校验为有效 Pygame 键码。校验失败记录 warning 并使用默认值。 |
| IP-002 | input_handler.py / boost key polling | internal | InputHandler 每帧通过 `pygame.key.get_pressed()[BOOST_KEY]` 轮询加速键状态。暴露 `is_boost_pressed() -> bool` 供 Game 调用。内部维护标志位防止焦点丢失后物理按键残留导致误激活（需释放后重新按下 KEYDOWN）。 |
| IP-003 | game.py / decoupled game loop | internal | Game.run() 解耦渲染与逻辑：渲染基于 `RENDER_FPS`（60fps），逻辑基于时间累加器 + 可变 tick_interval。每帧计算 `tick_interval = BASE_TICK_INTERVAL / snake.boost_multiplier`，累加 dt 后条件执行 `_update()`。包含 spiral-of-death 保护（每帧最多执行 max_catchup 步）。 |
| IP-004 | renderer.py / boost visual indicator | internal | Renderer.draw_frame() 接收 `is_boosting: bool` 参数，传递给各子绘制方法。`draw_snake()` 根据标志选择颜色，`draw_hud()` 根据标志渲染 "BOOST" 文字。`tick()` 使用 `RENDER_FPS` 替代原有 `FPS`。 |
| IP-005 | game.py / state transition reset | internal | Game.reset() 和状态变更为 GAME_OVER/VICTORY 时，调用 InputHandler 的 boost 状态复位（或 Game 直接设置 snake.boost_state 回默认值）。焦点丢失由 InputHandler.process_events() 检测 WINDOWFOCUSLOST 事件并复位内部 _boost_pressed 标志。 |
| IP-006 | snake.py / boost state query | internal | Renderer 和 Game 均通过 `snake.is_boosting` 和 `snake.boost_multiplier` 只读 property 查询加速状态。Snake 是 boost 状态的唯一数据源（Single Source of Truth）。 |

## 6. 设计决策

| 编号 | 决策 | 理由 | 备选方案 | 权衡 |
|------|------|------|---------|------|
| DD-001 | **解耦渲染帧率与逻辑 tick**：渲染固定在 RENDER_FPS（60fps），游戏逻辑按可变 tick_interval 独立推进 | (1) 加速只改变逻辑速度，视觉帧率保持流畅不跳变；(2) 支持平滑过渡——tick_interval 可连续变化，不依赖离散 FPS 值；(3) 为未来功能（暂停动画、粒子效果）打基础 | **方案 B**: 动态改变 FPS（boost 时 FPS *= multiplier）。优点：改动极小（改一行 tick 调用）。缺点：渲染帧率随之变化，视觉卡顿；无法实现平滑过渡 | 解耦方案改动较大（重构 run() 循环），但架构收益显著——逻辑和渲染彻底分离。一次性重构避免后续功能反复修补循环 |
| DD-002 | **boost 状态放在 Snake 实例属性** | (1) 满足约束 C-008（为多人扩展留余地——每条蛇独立加速状态）；(2) Snake 是碰撞检测和移动的承载者，加速状态与蛇本体紧密内聚；(3) Renderer 和 Game 均通过 `snake.is_boosting` 查询，接口清晰 | **方案 B**: boost 状态放在 Game 类。优点：Game 本来就需要它计算 tick_interval，少一层传递。缺点：违反 C-008，多人场景需要重构 | 当前单人场景下两者工作量相近。选择 Snake 遵循 C-008 并为扩展预留架构空间 |
| DD-003 | **加速键检测使用轮询 (`pygame.key.get_pressed()`)** 而非事件驱动 (`KEYDOWN`/`KEYUP`) | (1) 加速是持续 hold 状态，不是一次性事件——轮询天然支持"按住不放"语义；(2) 避免与操作系统 key-repeat 事件混淆（BC-009）；(3) `get_pressed()` 每帧返回物理按键状态，不受事件队列消费顺序影响 | **方案 B**: KEYDOWN 设置标志 + KEYUP 清除标志。优点：与现有方向键处理风格一致。缺点：需自行区分 key-repeat（OS 自动发送重复 KEYDOWN），增加去重逻辑；窗口失焦时 KEYUP 可能丢失 | 轮询方案的 minor 缺点：调用 `get_pressed()` 遍历全部键状态（~300 键），实际开销 < 0.1ms，可忽略。配合标志位解决焦点丢失后的残留问题 |
| DD-004 | **视觉指示采用蛇身颜色变更 + HUD 文字双通道** | (1) 颜色变更是最直观的反馈，玩家余光即可感知；(2) HUD 文字为冗余通道，满足 NFR-U003 色觉友好要求；(3) 仅使用现有渲染基础设施，零额外资源 | **方案 B**: 屏幕边框发光效果。优点：更醒目。缺点：实现复杂（需多次绘制边框或 shader），渲染开销较大。**方案 C**: 仅颜色变更。缺点：色觉障碍玩家无法感知 | 双通道方案实现简单（改颜色值 + 绘制一段文字），符合 FR-004 "至少实现一种"的要求；NFR-U003 要求冗余通道，双通道正好满足 |
| DD-005 | **平滑过渡使用线性插值 tick_interval**（而非线性插值 multiplier 再计算 interval） | (1) 直接插值 `current_multiplier`，等价于插值 interval；(2) 过渡时长固定为 BOOST_TRANSITION_SECONDS（约 150ms），符合 FR-007；(3) 插值增量 `(target - current) * dt / TRANSITION` 计算量极小 | **方案 B**: 分步切换（如每 50ms 调整 multiplier 0.25）。优点：更简单。缺点：阶梯感明显，体验不如连续过渡。**方案 C**: ease-in-out 曲线。优点：手感更自然。缺点：实现复杂，当前为 P2 需求不值得 | 线性插值在 150ms 短时过渡内与 ease-in-out 的体验差异微小。简单实现适配 P2 优先级。后期如需优化手感，将线性替换为缓动曲线代价极小 |
| DD-006 | **每帧最多执行有限次逻辑 catch-up**（spiral-of-death 保护），默认上限约 5 步/帧 | (1) 防止极端情况（如 tick_interval 极短或帧率骤降）导致一帧内执行数百次 `_update()` 造成假死；(2) 5 步/帧上限对应 300 步/秒（@60fps），远超 5x 加速的 50 步/秒，正常情况不会触发 | **方案 B**: 不做保护，完全信任 tick_interval 不会异常。缺点：参数配置错误或 bug 可能导致游戏卡死。**方案 C**: 单帧只执行一步，其余时间丢弃。缺点：速度超出上限时无 catch-up，游戏变慢 | 保护上限是防御性编程的标准做法，实现成本低（一个 while 循环计数器），避免参数异常导致不可恢复的假死 |
| DD-007 | **状态转换时 boost 状态由 Game.reset() 统一复位**，而非在各转换点分散处理 | (1) Game.reset() 已经统一管理 Snake/Food/Score/State 的全部复位，boost 复位是自然的扩展；(2) GAME_OVER/VICTORY 触发时在 `_update()` 内部设置 state，随后在 run() 的下一帧开始前 boost 复位；或直接在状态变更语句后立即复位 | **方案 B**: 在 Game._update() 中每个 state=GAME_OVER 出口后追加 boost_reset()。优点：即时复位。缺点：重复代码，容易遗漏 | 混合方案：`_update()` 中检测到 state 变更时立即复位 boost（保证碰撞告警），`reset()` 中对 boost 再次复位（保证重启一致性）。双重保险满足 NFR-R001 的 100% 复位要求 |

## 附录: 架构影响评估

### A.1 游戏循环重构风险

现有 `run()` 循环假设每帧执行一次 `_update()`（蛇移动一步）。解耦后 `_update()` 可能在单帧内执行 0 次或多步。影响：

- **渲染一致性**: `draw_frame()` 在每帧末尾对所有逻辑步的结果渲染一次，不存在"渲染到一半逻辑状态"的问题
- **输入一致性**: 输入处理每渲染帧执行一次（在逻辑更新前），单帧内多步逻辑使用同一输入命令——与现有行为语义一致
- **碰撞回归风险**: 多步 catch-up 时，`_update()` 的碰撞检测逻辑本身不变（逐步检测），但若 catch-up 步数过多可能有间歇性帧延迟。通过在 `_update()` 内维持不变的碰撞逻辑 + 每步独立检测，碰撞正确性不受影响

### A.2 未变更模块

- `food.py`: 无修改。食物生成逻辑与速度无关
- 所有测试模块 (`test_*.py`): 测试需要新增加速模式测试用例，但测试文件结构不变
