# 概要设计

> **功能**: 经典贪吃蛇游戏 (Snake Game) | **阶段**: 概要设计 (P2) | **角色**: 软件架构师 | **日期**: 2026-07-07

---

## 1. 架构模式

### 选择：Game Loop Pattern（游戏主循环模式）

### 理由

贪吃蛇属于典型的实时交互游戏，其核心运行机制天然符合 Game Loop 模式：每帧依次执行 **输入处理 -> 状态更新 -> 渲染绘制 -> 帧率控制**，形成一个持续运行的闭环。该模式是游戏开发领域的事实标准，Pygame 库本身的设计也围绕此模式展开。

与 MVC 或 ECS 等其他模式相比，Game Loop 的调用链路最直接：事件一帧内即被消费并反映到画面上，延迟路径最短，这对要求 200ms 内输入可见的实时游戏至关重要。

考虑到本项目规模小（约 300-400 行代码）、无网络层、无异步需求、无持久化存储，游戏循环内的子模块按职责划分为独立组件即可满足可维护性要求，无需引入 MVC 的分层间接性，也不必使用 ECS 的系统遍历开销。

## 2. 组件划分

### 2.1 组件总览

```
main.py (入口)
  └── Game (游戏协调器)
        ├── Config      (配置常量)
        ├── Snake       (蛇实体)
        ├── Food        (食物实体)
        ├── Renderer    (渲染器)
        └── InputHandler(输入处理器)
```

### 2.2 组件定义

| 组件 | 文件 | 职责 | 对外接口 | 依赖 |
|------|------|------|----------|------|
| **Config** | `config.py` | 集中管理所有可配置常量：窗口尺寸、网格参数、颜色定义、游戏速度、初始值等 | 模块级常量：`WINDOW_WIDTH`, `WINDOW_HEIGHT`, `CELL_SIZE`, `FPS`, `INITIAL_SNAKE_LENGTH`, `SCORE_PER_FOOD`, `COLORS` | 无 |
| **Snake** | `snake.py` | 管理蛇身坐标列表、移动方向；提供移动、增长、方向切换（含反向拦截）、自撞检测、边界碰撞检测 | `move()`, `grow()`, `change_direction(dx, dy)`, `check_boundary_collision(cols, rows) -> bool`, `check_self_collision() -> bool`, `head -> (x, y)`, `body -> List[(x, y)]`, `direction -> (dx, dy)`, `reset()` | 无（纯逻辑组件） |
| **Food** | `food.py` | 管理食物坐标；在未被蛇身占据的网格中随机生成食物位置，含边界兜底（全满则胜利） | `respawn(occupied_cells: List[(x,y)]) -> bool`, `position -> (x, y)`, `reset()` | `random` (Python stdlib) |
| **Renderer** | `renderer.py` | 封装所有 Pygame 绘制调用：背景填充、蛇身绘制（含蛇头颜色区分）、食物绘制、HUD 分数显示、游戏结束/胜利叠加层、绘制顺序管理 | `draw_background(screen)`, `draw_snake(screen, snake)`, `draw_food(screen, food)`, `draw_hud(screen, score)`, `draw_game_over(screen, score)`, `draw_victory(screen, score)`, `draw_frame(screen, snake, food, score, state)` | `pygame.display`, `pygame.draw`, `pygame.font`, `Config` |
| **InputHandler** | `input_handler.py` | 处理 Pygame 事件队列：捕获 QUIT 事件（窗口关闭）、捕获 KEYDOWN 事件（方向键/重新开始键）、反向输入拦截、焦点丢失检测（推荐暂停） | `process_events(snake_direction) -> dict` (返回 `{"action": "direction"\|"restart"\|"quit"\|"none", "direction": (dx,dy)\|None}`) | `pygame.event`, `pygame.locals` |
| **Game** | `game.py` | 游戏协调器：持有 Snake/Food/Renderer/InputHandler 实例，管理分数和状态机（RUNNING / GAME_OVER / VICTORY），编排主循环各阶段，控制逻辑帧率 | `run()`, `reset()` | `Snake`, `Food`, `Renderer`, `InputHandler`, `Config`, `pygame.time.Clock` |
| **main** | `main.py` | 入口点：初始化 Pygame，创建窗口，实例化 Game 并调用 `run()`，退出前调用 `pygame.quit()` | `main()` 函数 | `pygame`, `Game`, `Config` |

### 2.3 状态机

```
                  +---- R键/空格键 ----+
                  |                    |
                  v                    |
   [ 启动 ] -> [RUNNING] --撞墙/自撞--> [GAME_OVER]
                  |                        |
                  +--- 全板占满 ---------> [VICTORY]
                  |                        |
                  +--- 窗口关闭 ---------> [QUIT]

   RESET 转换时：Snake.reset(), Food.reset(), score=0
```

## 4. 数据流

### 4.1 主循环数据流

```
┌─────────────────────────────────────────────────────────┐
│                      Game.run()                          │
│                                                          │
│  1. clock.tick(FPS)         ← 控制帧率                   │
│  2. events = pygame.event.get()                         │
│  3. command = InputHandler.process_events(events,        │
│                               snake.direction)           │
│       │                                                  │
│       ├── action="direction" → snake.change_direction()  │
│       ├── action="restart"   → self.reset()              │
│       ├── action="quit"      → break loop                │
│       └── action="none"      → (continue)                │
│                                                          │
│  4. IF state == RUNNING:                                 │
│       a. snake.move()                                    │
│       b. IF snake.head == food.position:                 │
│            snake.grow()                                  │
│            score += SCORE_PER_FOOD                       │
│            IF NOT food.respawn(snake.body):              │
│                state = VICTORY                           │
│       c. IF snake.check_boundary_collision() OR          │
│             snake.check_self_collision():                │
│            state = GAME_OVER                             │
│                                                          │
│  5. Renderer.draw_frame(screen, snake, food,             │
│                          score, state)                   │
│       ├── draw_background()                              │
│       ├── draw_snake()        (蛇尾→蛇头，蛇头颜色不同)   │
│       ├── draw_food()                                    │
│       ├── draw_hud()          (分数)                     │
│       └── IF state == GAME_OVER: draw_game_over(score)   │
│           IF state == VICTORY:   draw_victory(score)     │
│                                                          │
│  6. pygame.display.flip()    ← 双缓冲交换，输出画面       │
└─────────────────────────────────────────────────────────┘
```

### 4.2 输入处理数据流 (InputHandler)

```
pygame.event.get()
  │
  ├── QUIT event → return {"action": "quit"}
  │
  ├── KEYDOWN + K_UP    → if current_dir != (0, 1):
  │                          return {"action": "direction",
  │                                  "direction": (0, -1)}
  ├── KEYDOWN + K_DOWN  → if current_dir != (0, -1):
  │                          return {"action": "direction",
  │                                  "direction": (0, 1)}
  ├── KEYDOWN + K_LEFT  → if current_dir != (1, 0):
  │                          return {"action": "direction",
  │                                  "direction": (-1, 0)}
  ├── KEYDOWN + K_RIGHT → if current_dir != (-1, 0):
  │                          return {"action": "direction",
  │                                  "direction": (1, 0)}
  │
  ├── KEYDOWN + K_r     → (仅 game_over/victory 状态)
  │                        return {"action": "restart"}
  │
  └── (其他事件)        → return {"action": "none"}

每帧仅最后处理的 direction 生效（单次方向切换；非完整命令队列）
```

### 4.3 蛇移动与碰撞数据流

```
snake.move()
  │
  ├── 计算新蛇头坐标:
  │   new_head = (body[0].x + direction.dx,
  │               body[0].y + direction.dy)
  │
  ├── 移动逻辑:
  │   body.insert(0, new_head)   # 头部插入
  │   body.pop()                 # 尾部移除（若无增长）
  │
  └── 注意: 先 insert 再 pop，grow() 时跳过 pop

snake.check_boundary_collision(cols, rows)
  │
  └── head = body[0]
      判断: head.x < 0 OR head.x >= cols
            OR head.y < 0 OR head.y >= rows

snake.check_self_collision()
  │
  └── head = body[0]
      判断: head in body[1:]    # O(n) 线性扫描
      注意: 仅检查移动后的状态，移动前重合由 move 前检查保证
```

## 6. 设计决策

### 决策 1：网格固定尺寸 20x20 像素，窗口 800x600（40x30 网格）

| 项目 | 内容 |
|------|------|
| **决策** | 窗口 800x600 像素，单元格 20x20 像素，形成 40x30 的网格游戏区域 |
| **理由** | 800x600 是经典桌面游戏窗口尺寸，视觉舒适；20x20 单元格在渲染蛇身和食物时有充足的像素作圆形/圆角矩形绘制，视觉区分度高；40x30=1200 个格子提供了足够大的游戏空间（最长蛇可达 1200 节），远超典型游戏时长所需的容量 |
| **备选方案** | 640x480+20x20(32x24=768格) — 格子偏少，高级玩家可能较快触及边界附近；640x480+16x16(40x30) — 单元格过小视觉辨识度下降 |
| **权衡** | 800x600 比 640x480 占用更多屏幕空间，但对现代显示器完全可接受；20x20 是整除性最好且视觉最优的选择 |

### 决策 2：逻辑更新帧率固定 10 FPS，渲染不设上限

| 项目 | 内容 |
|------|------|
| **决策** | 游戏逻辑以固定 10 FPS 更新（每 100ms 移动一步），渲染由 `clock.tick()` 控制但实际渲染频率跟随显示器刷新率 |
| **理由** | 需求明确要求固定速度、不随分数增长而加速。10 FPS 是贪吃蛇的经典节奏：既不过快导致操作困难，也不过慢导致无聊。渲染与逻辑解耦：`clock.tick(FPS)` 仅作为最小帧间隔，实际渲染在逻辑更新之间可多次执行，保证画面刷新流畅 |
| **备选方案** | 15 FPS — 对新手偏快；8 FPS — 可能感觉延迟；动态 FPS（随分数提升）— 需求明确排除 |
| **权衡** | 10 FPS 意味着每 100ms 仅一次逻辑更新，快速连按方向键时可能错过某一帧的方向切换。通过方向缓存（本帧最后按下的有效方向）缓解，而非增加 FPS |

### 决策 3：方向输入策略 — 每帧单方向缓存，非队列

| 项目 | 内容 |
|------|------|
| **决策** | 每逻辑帧内，InputHandler 处理所有积压的 KEYDOWN 事件，仅最后一个有效方向（通过反向检查）生效，其余丢弃。不维护方向命令队列 |
| **理由** | 需求 FR-001 要求方向键"立即切换"，E-004 要求处理快速连按场景。单方向缓存是经典的蛇游戏输入策略：避免玩家通过快速输入"绕过"格子（如 3 帧内按下 右->下->左，非队列模式只执行最后的左转，若左转有效则执行，否则保持原方向），行为更可预测。队列模式下 3 个方向都会依次执行，可能导致玩家预期外的自撞 |
| **备选方案** | 方向命令队列 — 可精细操控但增加自撞风险，不符合休闲游戏定位；实时方向（每 event 立即生效）— 与固定帧率矛盾（event 发生在帧中间，而移动在帧边界） |
| **权衡** | 单方向缓存意味着高 APM 玩家可能"浪费"中间按键，但这对 10 FPS 的游戏已足够，且大幅简化逻辑 |

### 决策 4：蛇身数据结构 — List[(x, y)]，头在前（索引 0）

| 项目 | 内容 |
|------|------|
| **决策** | 使用 Python `list` 存储蛇身坐标元组，索引 0 为蛇头，末尾为蛇尾。移动时 `insert(0, new_head)` + `pop()`，增长时只 `insert(0)` 不 `pop()` |
| **理由** | 蛇身长度通常不超过数百节。`list.insert(0)` 的 O(n) 开销在此量级可忽略（1200 节满屏时仅约 0.1ms）。自撞检测用 `head in body[1:]` 是 O(n)，同样可接受。此方案代码简洁，无需引入 `collections.deque` |
| **备选方案** | `collections.deque` — 双端插入 O(1)，但 `deque` 不支持切片和 `in` 操作同样为 O(n)，且代码可读性略低；链表 — 过度设计 |
| **权衡** | 若未来蛇长扩展到万级（如开放世界贪吃蛇），需改用 `set` 做碰撞检测或空间哈希。当前需求下 list 已足够 |

### 决策 5：食物生成算法 — 随机选取空白格子，兜底胜利判定

| 项目 | 内容 |
|------|------|
| **决策** | `Food.respawn()` 构建所有空白格子集合（全集 - 蛇身占据集），从中国随机选取一个。若空白集合为空（蛇身占满所有格子），返回 `False` 触发 VICTORY 状态 |
| **理由** | 需求 E-001 和 E-008 要求处理满屏和生成失败的场景。全集筛选取空白格子的方法保证 100% 不会选中蛇身占据的格子（含蛇头），且算法复杂度 O(n) 在 n=1200 时极快。明确返回布尔值让 Game 层决定状态转换，职责清晰 |
| **备选方案** | 随机坐标 + 重试循环 — 简单但存在理论上的无限循环风险（虽然实际概率极低） |
| **权衡** | 每次生成食物都需构建空白格子集合（集合差运算），在蛇身接近满屏时计算量略有增加，但 1200 元素的集合操作在 Python 中仍在微秒级 |

### 决策 6：源文件组织 — 多模块拆分，非单文件

| 项目 | 内容 |
|------|------|
| **决策** | 将游戏拆分为 7 个模块文件：`config.py`、`snake.py`、`food.py`、`renderer.py`、`input_handler.py`、`game.py`、`main.py`，统一放在项目根目录或 `src/` 目录下 |
| **理由** | 虽然总代码量约 300-400 行，单文件完全可行，但多模块拆分在架构层面带来明确收益：(1) 各组件职责边界清晰，需求映射关系明确；(2) 便于独立测试（如 Snake 纯逻辑可无 Pygame 环境下单元测试）；(3) 为未来扩展（如道具系统、关卡系统）留下清晰扩展点 |
| **备选方案** | 单文件 `snake_game.py` — 开发最快，约 200-300 行；虽功能相同但不利于架构文档化和后续扩展 |
| **权衡** | 多模块增加约 10-15 行导入代码，模块间耦合通过 Game 协调器管理。对小型项目这是可接受的代价 |

### 决策 7：渲染管线 — 按层绘制，蛇身分节渲染含蛇头区分

| 项目 | 内容 |
|------|------|
| **决策** | 每帧渲染顺序：背景填充 -> 蛇身（从尾到头，蛇头用不同颜色或高亮） -> 食物 -> HUD 分数 -> 游戏结束/胜利叠加层。Renderer 封装所有绘制调用，接收 screen surface 作为参数 |
| **理由** | 分层绘制避免叠加覆盖错误（如 HUD 被蛇身遮挡）。蛇头颜色区分帮助玩家快速定位蛇的朝向。Renderer 封装使得 Game 层不直接操作 Pygame 绘制 API，降低耦合 |
| **备选方案** | 蛇身和蛇头同色 — 视觉辨识度低；在 Game 层直接调用 pygame.draw — 耦合度高 |
| **权衡** | 额外的 Renderer 抽象层增加 1 个文件，但保持 Game 层仅关注游戏逻辑协调，不涉及渲染细节 |

### 决策 8：窗口焦点丢失处理 — 推荐暂停，降级为继续运行

| 项目 | 内容 |
|------|------|
| **决策** | 检测 Pygame 的焦点丢失事件（`WINDOWEVENT_FOCUS_LOST`），设置 `paused` 标志暂停逻辑更新。若因平台兼容性问题无法可靠检测焦点事件，降级为不暂停（游戏继续运行），确保不崩溃 |
| **理由** | E-005 要求窗口失焦时不应崩溃，推荐暂停。Pygame 提供了 `pygame.WINDOWEVENT_FOCUS_LOST` 事件供检测。暂停是更好的用户体验（避免回神时已死亡），但某些 Linux 窗口管理器下此事件可能不可靠，因此设计降级策略 |
| **备选方案** | 不处理焦点 — 最简单的实现，但用户体验较差；强制暂停 — 有兼容性风险 |
| **权衡** | 暂停/继续运行均可接受，设计上优先检测→暂停→降级继续运行，确保跨平台兼容性和稳定性 |

### 决策 9：依赖检查 — 启动时 try/except ImportError

| 项目 | 内容 |
|------|------|
| **决策** | `main.py` 中通过 `try: import pygame` 捕获 `ImportError`，输出清晰的中英文安装提示后退出 |
| **理由** | E-009 要求 Pygame 未安装时给出清晰提示。用 try/except 包裹是最简洁可靠的方案。提示信息包含 `pip install pygame` 命令，降低用户解决依赖问题的门槛 |
| **备选方案** | 预检查脚本 / `requirements.txt` — 辅助手段，但不能代替启动时的运行时检查 |
| **权衡** | 无实质权衡，这是 Python 脚本的标准依赖检查方式 |

## 附录 A: 配置常量清单 (config.py)

| 常量 | 建议值 | 说明 |
|------|--------|------|
| `WINDOW_WIDTH` | 800 | 窗口像素宽度 |
| `WINDOW_HEIGHT` | 600 | 窗口像素高度 |
| `CELL_SIZE` | 20 | 网格单元边长（像素） |
| `GRID_COLS` | 40 | 网格列数 (=800/20) |
| `GRID_ROWS` | 30 | 网格行数 (=600/20) |
| `FPS` | 10 | 逻辑更新帧率 |
| `INITIAL_SNAKE_LENGTH` | 3 | 蛇初始长度 |
| `SCORE_PER_FOOD` | 10 | 每个食物分数 |
| `WINDOW_TITLE` | "Snake Game" | 窗口标题 |
| `COLORS.BACKGROUND` | (0, 0, 0) | 背景色（黑） |
| `COLORS.SNAKE_BODY` | (0, 255, 0) | 蛇身颜色（绿） |
| `COLORS.SNAKE_HEAD` | (0, 200, 0) | 蛇头颜色（深绿） |
| `COLORS.FOOD` | (255, 0, 0) | 食物颜色（红） |
| `COLORS.TEXT` | (255, 255, 255) | 文字颜色（白） |
| `COLORS.OVERLAY` | (128, 128, 128) | 叠加层半透明背景 |

## 附录 B: 文件结构建议

```
snake/
├── main.py              # 入口, Pygame 初始化, 依赖检查
├── game.py              # Game 协调器, 主循环, 状态机
├── snake.py             # Snake 实体 (纯逻辑, 无 Pygame 依赖)
├── food.py              # Food 实体 (依赖 random)
├── renderer.py          # 渲染 (依赖 pygame + config)
├── input_handler.py     # 输入处理 (依赖 pygame)
├── config.py            # 所有配置常量
├── requirements.txt     # pygame>=2.0
└── docs/
    └── feature-snake-game/
        ├── 需求分析.md
        └── 概要设计.md
```
