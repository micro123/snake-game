# 概要设计

> **功能**: 分数递增难度（Score-Based Difficulty Scaling） | **阶段**: 需求与设计 | **角色**: software-architect | **日期**: 2026-07-10

## 1. 架构模式

### 选择：MVC 协调器模式（现存架构不变）

**理由**：现有代码以 `Game` 为协调器、`Snake`/`Food` 为模型、`Renderer` 为视图、`InputHandler` 为输入代理的类 MVC 结构清晰且足够灵活，难度功能无需引入新模式。

- `difficulty_multiplier` 作为游戏级状态（分数驱动）自然归属于 `Game` 协调器，与 `Snake` 的 `boost_state`（输入驱动）职责分离
- tick 间隔计算从单一 boost 因子扩展为复合 `(1 + diff + boost)` 因子，仅需修改一处核心公式（`_get_current_tick_interval()`）
- Renderer 的 HUD 扩展遵循现有图层叠加模式，无需重构渲染管线
- 配置沿用 `config.py` 模块级常量 + `_validate` 启动校验范式，零侵入

**核心公式**：

```
tick_interval = max(BASE_TICK_INTERVAL / (1 + difficulty_multiplier + boost_multiplier), 20.0)
```

## 2. 组件划分

| 组件 | 职责 | 接口 | 依赖 |
|------|------|------|------|
| `config.py` — DifficultyConfig 常量块 | 定义 `SCORE_THRESHOLD_INTERVAL`、`DIFFICULTY_INCREMENT`、`MAX_DIFFICULTY_MULTIPLIER` 三个模块级常量及对应校验逻辑；追加校验到 `_validate_boost_config()`（或重命名为 `_validate_config()`） | 模块级常量导入：`SCORE_THRESHOLD_INTERVAL: int`, `DIFFICULTY_INCREMENT: float`, `MAX_DIFFICULTY_MULTIPLIER: float` | 现有 `BASE_TICK_INTERVAL`（用于推导 `MAX_DIFFICULTY_MULTIPLIER` 合理上限） |
| `Game` — DifficultyState 状态管理 | 持有 `difficulty_multiplier` 和 `difficulty_level`；在 `_update()` 分数变更后同步更新难度；在 `_get_current_tick_interval()` 中复合叠加 difficulty + boost 两个因子；在 `reset()` 中清零难度状态 | `_update_difficulty(): None` — 根据 score 重新计算 level 和 multiplier；`_get_current_tick_interval(): float` — 扩展为 `BASE / (1 + diff + boost)`，floor=20ms；`reset(): None` — 扩展清零 `difficulty_multiplier` / `difficulty_level` | `config.py`（difficulty 常量）；`snake.Snake.boost_multiplier`（属性读取，叠加计算） |
| `Renderer` — `draw_hud()` 扩展 | 在现有 HUD 背景条上新增一行难度显示文字（格式：`Lv.N  Speed Mx`），接收 `difficulty_level` 和综合倍率参数 | `draw_hud(score, is_boosting, difficulty_level, effective_multiplier): None` — 扩展签名，渲染难度信息 | `config.py`（`COLORS`, `HUD_HEIGHT`, `WINDOW_WIDTH`）；现有 `font_small`（复用字体缓存） |
| `Game` — `draw_frame` 调用点适配 | 在 `run()` 主循环中调用 `draw_frame` 时传递 `difficulty_level` 和 `effective_multiplier` 给 Renderer | `run()` 中 `renderer.draw_frame(...)` 调用扩展参数 | `Renderer.draw_frame()`；`Game.difficulty_level` / effective multiplier |

## 4. 数据流

```
┌─────────────────────────────────────────────────────────────────────┐
│ 数据流 1：分数变化 → 难度更新                                         │
│                                                                     │
│  玩家吃食物                                                           │
│    ↓                                                                │
│  Game._update() → score += SCORE_PER_FOOD                            │
│    ↓                                                                │
│  Game._update_difficulty()                                           │
│    ↓                                                                │
│  difficulty_level = score // SCORE_THRESHOLD_INTERVAL                │
│    ↓                                                                │
│  difficulty_multiplier = min(level * DIFFICULTY_INCREMENT,           │
│                               MAX_DIFFICULTY_MULTIPLIER)             │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│ 数据流 2：主循环 tick → 速度计算                                       │
│                                                                     │
│  Game.run() → accumulator >= tick_interval ?                         │
│    ↓                                                                │
│  Game._get_current_tick_interval()                                    │
│    ↓                                                                │
│  effective = 1 + difficulty_multiplier + snake.boost_multiplier     │
│    ↓                                                                │
│  interval = max(BASE_TICK_INTERVAL / effective, 20.0)               │
│    ↓                                                                │
│  返回给累加器，控制逻辑更新频率                                          │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│ 数据流 3：主循环渲染 → HUD 更新                                        │
│                                                                     │
│  Game.run() → renderer.draw_frame(...)                                │
│    ↓                                                                │
│  draw_frame(snake, food, score, state, is_boosting,                  │
│             difficulty_level, effective_multiplier)                   │
│    ↓                                                                │
│  draw_hud(score, is_boosting, difficulty_level,                      │
│           effective_multiplier)                                       │
│    ↓                                                                │
│  渲染 "Lv.N  Speed Mx"（格式示例："Lv.3  Speed 1.3x"）                │
└─────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────┐
│ 数据流 4：游戏结束/重启 → 难度复位                                       │
│                                                                     │
│  Game.reset()                                                        │
│    ↓                                                                │
│  difficulty_multiplier = 0.0                                         │
│  difficulty_level = 0                                                │
│    ↓                                                                │
│  下一局从初始速度 100ms 开始                                           │
└─────────────────────────────────────────────────────────────────────┘
```

## 5. 集成点

| 名称 | 类型 | 文件 | 说明 |
|------|------|------|------|
| `Game._get_current_tick_interval()` | internal | `game.py` | **核心集成点**：将现有 `BASE / boost` 公式扩展为 `BASE / (1 + difficulty + boost)`。引入 `difficulty_multiplier` 变量，叠加后施加 `max(20ms)` 下限保护。不改任何调用方，零副作用。 |
| `Game._update()` -> `_update_difficulty()` | internal | `game.py` | 在 `score += SCORE_PER_FOOD`（吃食物成功时）之后同步调用 `_update_difficulty()` 重新计算难度。因为每次吃食只 +10 分，阈值跨度 50 分，大多数调用不触发实际倍率变更。 |
| `Game.reset()` | internal | `game.py` | 扩展重置逻辑：在现有 `score=0` / `state=RUNNING` / `_reset_boost()` 之后追加 `difficulty_multiplier=0.0` 和 `difficulty_level=0`。 |
| `Renderer.draw_hud()` 签名扩展 | internal | `renderer.py` | 新增 `difficulty_level: int` 和 `effective_multiplier: float` 两个参数，在 HUD 背景条上渲染难度信息。`draw_frame()` 同步扩展签名透传参数。注意 HUD 布局调整：新增文字位于分数右侧或分数下方，避免遮挡。 |
| `config.py` 常量扩展 | internal | `config.py` | 新增三个模块级常量 `SCORE_THRESHOLD_INTERVAL(=50)`, `DIFFICULTY_INCREMENT(=0.1)`, `MAX_DIFFICULTY_MULTIPLIER(=4.0)`。在 `_validate_boost_config()` 中追加校验逻辑（或提取为更通用的 `_validate_config()`）。 |

## 6. 设计决策

| 决策 | 理由 | 备选方案 | 权衡 |
|------|------|----------|------|
| **DifficultyState 归属于 Game 协调器，不放在 Snake** | 难度倍率是分数驱动的游戏级状态，与 Snake 的输入驱动 `boost_state` 分属不同职责域。Game 已经管理 `score`、tick 计算和 `reset`，自然拥有 `difficulty_multiplier`。避免 Snake 依赖 score（循环依赖风险），保持 Snake 纯逻辑实体。 | (1) 放在 Snake 中：需要 Snake 感知 Game.score，引入不必要的耦合；Snake 的 reset 需要外部告知是否同时清零难度。 | Game 类略有膨胀（新增 2 属性 + 1 方法），但职责内聚更好，修改半径更小。 |
| **difficulty_multiplier 与 boost_multiplier 加法叠加（1 + diff + boost）** | 加法叠加更可控：player 可清晰感知两部分贡献。上限 clamp 简单：`总倍率 = min(1+diff+boost, 1+MAX_DIFF+MAX_BOOST)`。乘法叠加在两端最大时倍率膨胀过快（(1+4.0)*5.0 = 25x），会过早触及 tick 下限。 | (1) 乘法叠加：有效倍率 = (1+diff)*boost，两端最大可达 25x，远超过设计意图；(2) 取最大值：有效倍率 = 1+max(diff, boost)，但失去叠加感，不符合用户故事要求。 | 加法下 boost 在低分段效果更显著（叠加在较小基数上），但整体更安全、更直观。 |
| **difficulty_multiplier 使用 `level * increment` 公式计算，非逐次累加** | 避免 IEEE 754 浮点累加误差：0.1 累加 100 次得 10.000000000000002。基于 score 计算 level 再乘 increment 是幂等的，不存在误差累积。reset 后无需担心精度残留。 | (1) 每次跨越阈值时 `multiplier += DIFFICULTY_INCREMENT`：简单但有累积精度风险；(2) 使用 `decimal.Decimal`：精确但引入非标准库依赖，与现有 float 体系不一致。 | 乘法公式依赖阈值间隔固定不变，但若将来改为非线性难度曲线需重构；当前离散台阶式需求下此方案最优。 |
| **Renderer.draw_hud() 复用现有 `font_small`，不创建新 Font 对象** | FreeType Font 对象创建有开销（加载字体文件、解析字形）。难度显示使用与分数相同字号（28pt），完全可复用。新创建 Font 对象无收益仅增加内存占用和启动时间。 | (1) 新建 `font_difficulty`（独立字号 24pt）：更突出，但每次游戏启动多一次字体初始化；(2) 新建 font_difficulty 样式（粗体/斜体）：增强视觉区分但引入额外 Font 对象。 | 难度文字与分数共字号可能视觉区分度稍弱，但通过格式 `Lv.N` 前缀和适当间距足以区分。 |
| **难度信息仅在 HUD 中显示等级编号和综合倍率，不额外显示分数阈值进度** | 需求指定 HUD 格式为 `Lv.3 Speed 1.3x`，未要求显示进度条或"下一级还需X分"。保持信息简洁不增加玩家认知负担。如需更详细反馈可在后续迭代中扩展。 | (1) 显示进度 "Next Lv. in 30pt"：信息更丰富但 HUD 拥挤；(2) 仅显示等级编号不显示倍率：更简约但玩家无法直观感知速度变化幅度。 | 适中的信息密度，满足核心需求，为后续迭代留扩展空间。当前 HUD 高度 40px 容纳一行额外文字绰绰有余（分数左对齐 + 难度居中或右侧对齐）。 |
| **config.py 不新增独立 DifficultyConfig 类，沿用模块级常量 + `_validate` 函数范式** | 现有 config.py 的图案是模块级常量定义 + `_validate_boost_config()` 导入时自动校验。新增难度常量应遵循同一范式，保持代码风格一致。DifficultyConfig 类会破坏 config.py 的扁平结构，且无需实例化。 | (1) 创建独立 `DifficultyConfig` 数据类：更结构化但引入新模式，与现有风格冲突；(2) 难度常量放在 `game.py` 中：未复用配置集中管理模式，修改不便。 | constants-in-namespace 方式足够简洁，与现有代码风格一致，有经验的开发者可直接理解。建议将 `_validate_boost_config()` 重命名为 `_validate_config()` 以准确反映扩展后的职责范围。 |
