> **功能**: 分数递增难度 (Score-Based Difficulty Scaling) | **阶段**: 评估与排期 (P4) | **角色**: 项目经理 | **日期**: 2026-07-10

## 1. 难度评估

### 1.1 逐模块难度评分

| 模块 | 评分(1-10) | 理由 |
|------|-----------|------|
| config.py — 难度常量及校验 | 2 | 新增 4 个模块级常量 + 1 个 `_validate_difficulty_config()` 函数，完全遵循现有 `_validate_boost_config()` 的 global 复位 + `logging.WARNING` 模式。`MIN_TICK_INTERVAL` 从 game.py 硬编码 20 提取为显式常量，属于安全重构。无新架构模式，零风险。 |
| Game — difficulty_state 状态管理 + tick 公式 | 3 | 核心集成点。`_get_current_tick_interval()` 公式从 `BASE/boost` 扩展为 `BASE/(1+diff+(boost-1))`，数学上在 `diff=0` 时与原公式等价（向后兼容）。需正确处理 `boost_multiplier` 语义（属性已保证 `>=1.0`，但 `-1.0` 提取超出部分的逻辑必须精确）。`_update_difficulty()` 本身简单（`score//threshold * increment + clamp`），但 `_update()` 中调用点需放在 score 增量之后、食物重新生成之前。`reset()` 需追加两行复位。`run()` 需新增 `effective_multiplier` 局部计算并传递。总计约 45 行修改，涉及 5 个方法。 |
| Renderer — HUD 难度显示 | 2 | `draw_hud()` 和 `draw_frame()` 各扩展两个带默认值的参数（向后兼容），在 HUD 背景条上新增一行居中文字。复用现有 `font_small`，不创建新 Font 对象。布局为三部分式：左 Score \| 中 Lv/Speed \| 右 BOOST(条件)，极端值下可能存在轻微重叠，但格式固定长度短（`Lv.999 Speed 20.9x` 约 200px），800px 窗口宽度充裕。总计约 15 行修改。 |
| 测试套件适配 | 2 | 现有 8 个测试文件覆盖全面。`draw_hud`/`draw_frame` 签名的默认参数保证现有测试零断裂。需为 `_update_difficulty()` 新增测试类（约 8-10 个用例），为难度叠加 tick 计算新增测试（约 5 个用例），为 `reset()` 新增难度复位断言（约 3 个用例），为 HUD 难度渲染新增测试（约 5 个用例）。总计约 80-120 行测试代码，遵循现有 unittest 模式。 |

### 1.2 整体复杂度

| 维度 | 结论 |
|------|------|
| **总体等级** | **Medium（中等）** |
| **详细说明** | 整体复杂度中等，主要原因：(1) `_get_current_tick_interval()` 是核心风险点，公式变更需 100% 保证 `diff=0` 时向后兼容；(2) 虽然每个模块改动量小（约 80 行生产代码），但改动分散在 Game 类的 5 个方法中，验证面较广；(3) tick 间隔计算的正确性影响整个游戏节奏，需要组合维度测试（`diff x boost` 所有排列）。加分项：设计文档决策质量高（加法叠加、幂等公式、默认参数向后兼容），显著降低了实现风险。 |

### 1.3 优先级

| 维度 | 结论 |
|------|------|
| **优先级** | **P2 — 应优先安排** |
| **理由** | 用户故事中该功能标记为 must-have，解决的是游戏体验随着局数增长而单调化的真实问题。从技术角度看，该功能改动量小、风险可控、无外部依赖，是典型的「高价值低风险」功能。不阻塞其他功能或发布，但作为当前项目的主要功能迭代应优先安排。 |

## 5. 建议实施方法

1. **config.py**：新增 `SCORE_THRESHOLD_INTERVAL(50)`、`DIFFICULTY_INCREMENT(0.1)`、`MIN_TICK_INTERVAL(20)`、`MAX_DIFFICULTY_MULTIPLIER(4.0)` 四个常量 + `_validate_difficulty_config()`，在 `_validate_boost_config()` 末尾调用。提取 `game.py` 中硬编码的 20 为 `MIN_TICK_INTERVAL` 常量引用。（零运行时影响，安全第一步）
2. **Game**：新增 `difficulty_level` / `difficulty_multiplier` 属性（`__init__`），实现 `_update_difficulty()`（`score // threshold * increment + clamp`），修改 `_get_current_tick_interval()` 公式为 `BASE/(1+diff+(boost-1))` 并在 exception handler 中改用 `MIN_TICK_INTERVAL` 常量，`_update()` 中在 `self.score += SCORE_PER_FOOD` 之后调用 `_update_difficulty()`，`reset()` 追加难度清零，`run()` 计算 `effective_multiplier` 并传递。（核心变更，每个变更点独立可测）
3. **Renderer**：扩展 `draw_hud(score, is_boosting=False, difficulty_level=0, effective_multiplier=1.0)` 和 `draw_frame(... difficulty_level=0, effective_multiplier=1.0)`，在 HUD bar 居中位置使用现有 `font_small` 渲染 `Lv.{level}  Speed {m}x`。（参数默认值保证向后兼容）
4. **测试**：运行现有测试套件确认零回归 → 为 `_update_difficulty()` 和难度叠加 tick 计算新增测试类 → 更新 reset 测试追加 difficulty 断言 → 新增 HUD 难度渲染测试 → 手动验收测试（0/50/100/200 分各档 + boost 叠加 + 重启验证）。
5. **验收**：`main.py` 不需要修改；启动游戏 → 吃 5 个食物(50 分) → 验证 tick 间隔从 100ms 缩短到约 90.9ms(1.1x) → 按住空格 → 验证叠加后约 47.6ms(2.1x) → 撞墙 → 验证 HUD 难度复位 → 重启再次验证。

**说明**：7 个任务分布于 4 个里程碑，改动局限在 4 个文件（`config.py` / `game.py` / `renderer.py` / 测试文件），零新文件、零新依赖。每个任务验收标准明确，依赖链清晰无循环。所有技术风险均已识别并有针对性的缓解措施。设计文档中加法叠加公式、幂等设计、默认参数向后兼容等决策为实施提供了坚实保障。

## 12. 附录：技术决策回顾

以下设计决策为本评估提供了依据，确认后不再重复讨论：

| 决策 | 内容 | 影响 |
|------|------|------|
| 加法叠加 | 公式 `BASE / (1 + diff + (boost - 1))`，diff=0 时退化为原公式 | 保证向后兼容，降低回归风险 |
| 幂等公式 | `_update_difficulty()` 基于 score 和 threshold 的确定函数 | 无副作用，任意次数调用结果一致 |
| 默认参数 | `draw_hud()` 和 `draw_frame()` 新参数带默认值 | 现有调用方零断裂 |
| 常量提取 | `MIN_TICK_INTERVAL` 从硬编码提升为显式常量 | 消除魔法数字，便于校验 |
| 自动校验 | `_validate_difficulty_config()` 遵循现有 `_validate_boost_config()` 模式 | 风格一致，易维护 |
