# 需求分析

> **功能**: 蛇移动速度加速 (Snake Speed Boost on Key Hold) | **阶段**: 需求分析 (P1) | **角色**: requirements-analyst + requirements-engineer | **日期**: 2026-07-08

## 1. 问题陈述与目标用户

### 1.1 问题陈述

当前贪吃蛇游戏以单一恒定速度运行，玩家无法主动调控游戏节奏。在蛇身较短、风险较低的前期阶段，玩家只能被动等待蛇按默认速度爬行，体验单调。缺乏速度调控机制降低了游戏的互动性与可重玩性。为玩家提供按键按住加速能力，可赋予玩家对游戏节奏的自主权，提升参与感并让熟练玩家快速跳过低风险阶段。

### 1.2 目标用户

所有贪吃蛇游戏玩家。主要受益者为希望加快游戏节奏的熟练玩家，以及在蛇身较短的低风险前期阶段想要快速推进的玩家。休闲玩家也可利用加速功能获得短时高强度的挑战体验。

### 1.3 用户故事

| ID | 角色 | 需求 | 目的 | 优先级 |
|----|------|------|------|--------|
| US-001 | 玩家 (player) | 按住指定的加速键提升蛇的移动速度 | 在低风险时段加速游戏进程 | must-have |
| US-002 | 玩家 (player) | 释放加速键后速度立即恢复正常 | 精确控制加速的激活时机 | must-have |
| US-003 | 玩家 (player) | 加速激活时有清晰的视觉指示（如蛇身颜色变化、发光效果或HUD文字） | 始终可知加速是否处于激活状态，无需猜测 | should-have |
| US-004 | 玩家 (player) | 速度过渡应平滑而非瞬间跳变（渐进加速与减速） | 游戏手感精良，避免突然的速度变化导致操作失误 | nice-to-have |
| US-005 | 玩家 (player) | 加速键可配置且与方向键分离 | 在切换方向时不误触加速 | nice-to-have |

## 2. 范围定义

### 2.1 包含范围

- 按键按住检测，实现临时速度加速（按住加速，释放恢复）
- 在 config.py 中定义可配置的速度倍率
- 加速状态的视觉指示（蛇身颜色变化、HUD文字或屏幕边框效果）
- 所有状态转换时速度重置为基准：game over、restart、pause 和窗口焦点丢失
- 非游玩状态（暂停、game over、未开始）中加速功能禁用

### 2.2 排除范围

- 永久速度升级或游戏场景中拾取的持续性速度道具
- 基于分数或蛇身长度自动调整难度的速度缩放
- 多级加速或可变加速等级
- 多人模式的速度机制
- 加速使用的冷却计时器或体力/资源限制系统
- 加速状态的音频反馈（音效）

## 3. 功能性需求

### FR-001 速度加速激活 (Speed Boost Activation)
- **描述**: 玩家按住指定加速键时，蛇的移动速度按照可配置的倍率（默认 2x）提升。加速仅影响游戏 tick 间隔（两次连续蛇步之间的时间），不影响单步网格距离。加速键必须独立于方向键，不干扰方向输入。
- **优先级**: P0
- **验收标准**:
  - 按住加速键期间，蛇的实际 tick 间隔 = BASE_TICK_INTERVAL / speed_multiplier，蛇步频率明显加快。
  - 加速期间蛇每次移动的网格距离保持为一个单元格，不会跳跃多个格。
  - 同时按住方向键和加速键，蛇同时进行方向输入处理和加速，两者不互相阻塞。

### FR-002 速度加速停用 (Speed Boost Deactivation)
- **描述**: 玩家释放加速键后，蛇的移动速度立即恢复为基准速度。释放响应应在下一次 tick 之前生效，不给玩家造成"速度残留"的错觉。
- **优先级**: P0
- **验收标准**:
  - 释放加速键后，下一个游戏 tick 的间隔恢复为 BASE_TICK_INTERVAL。
  - 从释放到恢复的延迟不超过一帧（< 16ms @ 60fps）。
  - 单次按下并快速释放（tap 而非 hold）不会导致加速状态闪烁或 tick 间隔抖动。

### FR-003 可配置速度倍率 (Configurable Speed Multiplier)
- **描述**: 速度倍率（加速倍率）和基准 tick 间隔作为常量定义在 config.py 中，便于调整和测试。倍率需定义合理上限以防止性能问题。
- **优先级**: P0
- **验收标准**:
  - config.py 中包含 BOOST_SPEED_MULTIPLIER（默认值 2.0）和 BASE_TICK_INTERVAL（沿用现有值）常量。
  - config.py 中包含 MAX_BOOST_MULTIPLIER（默认值 5.0）常量，用于参数校验。
  - 修改 config.py 中的倍率值后无需改动其他源代码即可生效。

### FR-004 加速状态视觉指示 (Boost State Visual Indicator)
- **描述**: 加速激活时，必须提供清晰的视觉反馈让玩家感知加速状态。建议实现方式包括：蛇身颜色变化（如色调偏移、亮度提升）、HUD 文本显示（如 "BOOST" 文字）、或屏幕边框效果。
- **优先级**: P1
- **验收标准**:
  - 加速激活时，视觉指示在 200ms 内出现。
  - 加速停用时，视觉指示在同一 tick 内消失。
  - 视觉指示在 800x600 窗口内清晰可见，不遮挡蛇身、食物和分数显示。
  - 至少实现一种视觉指示形式，具体形式在概要设计阶段确定。

### FR-005 状态转换时速度重置 (Speed Reset on State Transitions)
- **描述**: 当游戏进入 game over、restart、或 pause 状态时，或窗口失去焦点时，加速状态必须自动复位为基准速度（即释放加速效果），无论玩家是否仍按着加速键。
- **优先级**: P0
- **验收标准**:
  - 加速状态下蛇撞墙导致 game over：游戏结束，加速状态清除，重新开始后以基准速度运行。
  - 加速状态下按 P 键暂停：游戏暂停，加速状态清除，恢复后以基准速度运行（即使加速键仍被按住，需先释放再按才能重新激活加速）。
  - 加速状态下按 R 键重启：游戏重启，加速状态清除。
  - 加速状态下窗口失去焦点（如 Alt+Tab）：加速状态清除，重回窗口后以基准速度运行。
  - 游戏未开始（初始画面）或 game over 画面时，按住加速键不产生任何效果。

### FR-006 非游玩状态加速禁用 (Boost Disabled in Non-Playing States)
- **描述**: 加速功能仅在游戏处于"游玩中"状态时可用。在暂停、game over、未开始等状态下，加速输入被忽略。
- **优先级**: P0
- **验收标准**:
  - 暂停状态下按住加速键：恢复游戏时蛇以基准速度运行，不自动进入加速。
  - Game over 画面下按住加速键：无任何视觉效果或速度变化。
  - 初始画面（未开始）下按住加速键：无任何视觉效果或速度变化。

### FR-007 平滑速度过渡 (Smooth Speed Transition)
- **描述**: 速度从基准到加速（ramp-up）以及从加速到基准（ramp-down）的过渡应该是渐进的（例如在 100-200ms 内线性插值），而非瞬间切换，以提升操作手感并避免突兀的速度变化导致玩家判断失误。
- **优先级**: P2
- **验收标准**:
  - 按下加速键后，tick 间隔在约 150ms 内从 BASE_TICK_INTERVAL 平滑过渡到 BOOST_TICK_INTERVAL。
  - 释放加速键后，tick 间隔在约 150ms 内从 BOOST_TICK_INTERVAL 平滑恢复到 BASE_TICK_INTERVAL。
  - 过渡期间的 tick 间隔计算无除零错误或负值。
  - 如果平滑过渡与 FR-001/FR-002 的即时性要求冲突，以 FR-001/FR-002 为准——即最终加速/减速效果达成即可，过渡时间允许微调。

### FR-008 可配置加速键 (Configurable Boost Key)
- **描述**: 加速键绑定作为常量定义在 config.py 中，与方向键（Up/Down/Left/Right）及其他功能键（R-重启、Q-退出、P-暂停）不冲突。
- **优先级**: P2
- **验收标准**:
  - config.py 中包含 BOOST_KEY 常量（默认值为 pygame.K_SPACE）。
  - 修改 BOOST_KEY 后无需改动其他源代码即可生效。
  - BOOST_KEY 不与现有的方向键或功能键重叠。

### FR-009 方向与加速输入独立 (Independent Direction and Boost Input)
- **描述**: 玩家在按住加速键的同时切换方向，加速效果不应中断；同理，加速键的按下与释放不影响方向输入的处理。
- **优先级**: P0
- **验收标准**:
  - 加速期间按方向键改变方向：蛇转向成功且加速保持活跃。
  - 加速期间快速连续切换方向（如 Up -> Left -> Down）：加速不中断，方向切换不丢帧。
  - 方向切换帧内同时按下/释放加速键：两个行为各自独立生效，无竞态条件。

### FR-010 加速状态下碰撞完整性 (Collision Integrity at Boost Speed)
- **描述**: 在加速状态下，所有碰撞检测（墙壁碰撞、自身碰撞）和食物消费逻辑必须正确工作，不因 tick 间隔缩短而产生遗漏。
- **优先级**: P0
- **验收标准**:
  - 加速状态下蛇头碰到墙壁：立即触发 game over，不会穿透墙壁。
  - 加速状态下蛇头碰到自身身体：立即触发 game over。
  - 加速状态下蛇头经过食物所在格：食物被正常消费，蛇身增长，分数增加，食物重新生成。
  - 即使以最大倍率（5x）运行，连续 100 次食物消费后无碰撞遗漏或 false negative。
  - 现有碰撞相关的单元测试在加速模式下全部通过。

## 4. 非功能性需求

### 4.1 性能

- **NFR-P001**: 游戏在基准速度和最大加速倍率（5x）下均保持稳定帧率（≥55fps @ 800x600），帧率波动不超过 10%。
- **NFR-P002**: 加速状态下的 tick 间隔计算耗时不超过 1ms，不阻塞主循环。
- **NFR-P003**: 加速状态视觉指示的渲染开销不超过基准渲染的 5%（额外绘制调用可控）。
- **NFR-P004**: 按住加速键期间，CPU 使用率增加不超过 5%（避免键盘轮询或重绘带来的性能退化）。

### 4.2 安全

- **NFR-S001**: 加速倍率参数在加载时必须校验：若配置值超出 [1.0, MAX_BOOST_MULTIPLIER] 范围，自动 clamp 到有效边界并记录警告，不允许崩溃。
- **NFR-S002**: tick 间隔计算结果必须校验：不允许负值、零值或 NaN，出现异常值时回退到 BASE_TICK_INTERVAL。
- **NFR-S003**: 加速键绑定的键码必须在 Pygame 有效键码范围内，无效时回退到默认键（K_SPACE）并记录警告。

### 4.3 可用性

- **NFR-U001**: 首次使用加速功能的玩家应能自然发现加速键——建议在游戏 HUD 或初始画面中显示加速键提示文本（如 "Hold SPACE to boost"）。
- **NFR-U002**: 加速键默认绑定到空格键（物理位置居中，易于触及），且独立于方向键，避免误触。
- **NFR-U003**: 加速视觉指示不得使用仅依赖颜色传达信息的设计（考虑色觉障碍玩家），建议同时使用文字/图标辅助。
- **NFR-U004**: 加速键按住期间，手指疲劳度应在可接受范围内——默认空格键或 Shift 键属于大键面长键程按键，适合持续按压。

### 4.4 可靠性

- **NFR-R001**: 加速状态在任何游戏状态转换（game over、restart、pause、focus loss）后必须 100% 复位，不允许出现"加速粘滞"（stuck in boost）。
- **NFR-R002**: 窗口失去焦点时加速状态必须可靠清除——通过 pygame.WINDOWEVENT_FOCUS_LOST 事件检测。
- **NFR-R003**: 连续按住加速键 60 秒以上，游戏不出现内存泄漏、tick 漂移或速度异常。
- **NFR-R004**: 在加速键被操作系统/输入法拦截（如空格键在中文输入法中可能触发选字）的场景下，游戏需优雅处理——如果 Pygame 层面无法接收到按键事件则属于平台限制，但不得因输入法问题导致崩溃。

## 5. 约束条件

- **C-001**: 必须兼容现有 Pygame 事件循环架构，不引入新的第三方依赖。
- **C-002**: 加速实现必须基于调整游戏 tick 间隔（每次蛇移动之间的等待时间），不得修改蛇每次移动的网格步长。
- **C-003**: 窗口尺寸（800x600）和网格单元大小在加速功能中保持不变。
- **C-004**: 加速功能不得影响分数计算逻辑——分数仅由食物消费触发，与速度无关。
- **C-005**: 加速键必须作为新常量加入 config.py，不得硬编码键值于游戏逻辑代码中。
- **C-006**: 本功能不包含音频反馈（音效），视觉指示作为主要的加速状态反馈通道。
- **C-007**: 不得修改现有方向键绑定（Up/Down/Left/Right）和其他功能键（R-重启、Q-退出、P-暂停）。
- **C-008**: 加速功能仅作用于单人模式；本项目当前无多人模式，但设计上不应阻碍未来多人扩展（即加速状态应作为蛇的实例属性而非全局单例）。

## 6. 前提与假设

- **A-001**: 加速键是独立于四个方向键的专用按键。默认使用空格键，作为新常量在 config.py 中定义（BOOST_KEY）。假设空格键不与现有控制键冲突。
- **A-002**: 游戏主循环使用可配置的 tick 间隔（如 BASE_TICK_INTERVAL 在 config.py 中定义），加速时该间隔除以速度倍率以实现加速。
- **A-003**: 输入处理器能区分按键按住（持续按压）与按键轻触（短暂按压），或此区分逻辑可在本功能范围内实现。建议采用 `pygame.key.get_pressed()` 轮询方式检测加速键状态。
- **A-004**: 按住加速键不会消耗食物、触发游戏操作或以速度变化之外的方式影响分数计算。
- **A-005**: 视觉指示使用现有渲染基础设施；不需要新的渲染框架或外部资源文件。
- **A-006**: 游戏窗口尺寸（800x600）和网格维度在加速功能中保持不变；加速不改变空间层面的游戏玩法。
- **A-007**: 加速键与现有控制键无冲突：BOOST_KEY 不与方向键（Up/Down/Left/Right）、重启键（R）、退出键（Q）或暂停键（P）重叠。

## 7. 依赖项

- **D-001 Pygame 键盘事件处理**: 可通过 `pygame.key.get_pressed()` 进行连续按键状态轮询，或通过 KEYDOWN/KEYUP 事件处理。当前项目已通过 input_handler.py 具备基础键盘事件处理能力。
- **D-002 游戏循环计时机制**: 现有的帧率控制（renderer.py）和游戏 tick 间隔管理（game.py）必须支持运行时动态调整。加速功能依赖此机制的可动态修改性。
- **D-003 config.py**: 配置常量模块，需扩展以包含加速键绑定（BOOST_KEY）、速度倍率（BOOST_SPEED_MULTIPLIER）、最大倍率（MAX_BOOST_MULTIPLIER）和视觉指示参数。
- **D-004 渲染层**: 现有渲染层必须支持额外的视觉叠加或颜色修改，以显示加速状态指示。需要在 renderer.py 中新增或扩展渲染接口。

## 8. 数据实体

### DE-001 SpeedConfig
- **说明**: 与加速功能相关的配置参数集合，定义于 config.py。
- **字段**:
  - `BOOST_KEY`: int — Pygame 键码，默认 pygame.K_SPACE
  - `BOOST_SPEED_MULTIPLIER`: float — 加速倍率，默认 2.0
  - `MAX_BOOST_MULTIPLIER`: float — 允许的最大加速倍率，默认 5.0
  - `BASE_TICK_INTERVAL`: int — 基准 tick 间隔（毫秒），沿用现有值（预估 100-200ms）
  - `BOOST_TRANSITION_FRAMES`: int — 平滑过渡帧数（仅 FR-007 实现时需要），默认 6-9（对应 100-150ms @ 60fps）
- **关系**: 被 Game 对象在初始化时读取，用于计算实际 tick 间隔。

### DE-002 BoostState
- **说明**: 运行时加速状态，封装在 Snake 或 GameState 对象中。
- **字段**:
  - `is_active`: bool — 加速是否当前生效
  - `current_multiplier`: float — 当前实际加速倍率（平滑过渡时为中间值，即时模式下为 1.0 或 BOOST_SPEED_MULTIPLIER）
  - `transition_progress`: float — 过渡进度 0.0 ~ 1.0（仅平滑过渡时需要）
- **关系**: 由 Game 或 Snake 持有；在每帧 tick 计算时被查询以确定实际 tick 间隔。

### DE-003 KeyBinding（概念实体）
- **说明**: 加速键绑定的逻辑映射；当前仅涉及单一加速键，且在 config.py 中定义，无需运行时实体。若未来扩展为可重映射键位，则扩展此概念。
- **字段**:
  - `boost_key`: int — 加速键的 Pygame 键码
- **关系**: 与现有方向键绑定（UP_KEY, DOWN_KEY, LEFT_KEY, RIGHT_KEY）并列定义。

## 9. 边界情况

- **BC-001 加速中触发 Game Over**: 蛇在加速状态下撞墙或撞自身导致 game over，加速效果立即清除。Game over 画面下即使玩家仍按住加速键，不应有任何加速效果残留。重新开始后以基准速度运行。
- **BC-002 加速中暂停与恢复**: 加速状态下按暂停键，游戏暂停且加速清除。恢复后即使玩家未释放加速键，也不自动恢复加速——玩家必须释放后重新按下才能激活。这避免了"恢复即冲刺"的意外情况。
- **BC-003 游戏未开始时按住加速键**: 在初始画面或 game over 画面按住加速键，然后按 R 键开始游戏——游戏应以基准速度开始，不自动进入加速状态。本次按键按下不触发加速，需释放后重新按下。
- **BC-004 窗口失去焦点时加速键被按住**: 玩家按住加速键的同时 Alt+Tab 切换窗口，加速状态清除。切回游戏窗口后，即使加速键仍被物理按住，加速不激活——因为 Pygame 丢失了按键按下事件，需释放后重新按下产生新的 KEYDOWN 事件。
- **BC-005 快速连续按压释放**: 玩家在极短时间内（< 100ms）反复按下和释放加速键（连打/抖动），加速状态不应出现闪烁或导致 tick 间隔剧烈抖动。若实现平滑过渡，应以最后一个稳定状态为准。
- **BC-006 极端加速倍率**: 若配置 BOOST_SPEED_MULTIPLIER = 25（超过 MAX_BOOST_MULTIPLIER 限制），系统应 clamp 到 5.0 并记录警告，tick 间隔不低于 20ms，避免因单帧内多次蛇步导致的逻辑错误。
- **BC-007 食物恰好在加速开始/结束时生成**: 加速状态的切换与食物生成属于不同逻辑域，互不影响。食物在加速期间正常消费，在速度恢复期间正常消费，无关联时序问题。
- **BC-008 加速键与方向键同时按下**: 同一帧内方向键和加速键同时被按下，两个输入各自处理，不产生优先级冲突。方向变更和加速激活在同一 tick 或相邻 tick 内各自生效。
- **BC-009 操作系统键盘重复事件**: 操作系统层面的按键重复（key repeat）可能产生多次 KEYDOWN 事件。输入处理必须区分"按住加速键"（基于 pygame.key.get_pressed() 轮询或标志位）与"重复 KEYDOWN 事件"，避免重复触发加速状态切换逻辑。
- **BC-010 最大倍率下的自碰判断**: 以 5x 倍率运行时，tick 间隔缩短至基准的 1/5，蛇可能在一次转弯后快速接近自身身体。碰撞检测必须逐 tick 执行，确保不会因间隔过短而跳过蛇头与身体重叠的帧。

## 10. 集成点

- **IP-001 config.py**: 读取加速相关配置常量（BOOST_KEY, BOOST_SPEED_MULTIPLIER, MAX_BOOST_MULTIPLIER, BASE_TICK_INTERVAL）。在模块加载时进行参数校验。
- **IP-002 input_handler.py / 键盘事件处理**: 检测加速键的按下（KEYDOWN）和释放（KEYUP）事件，或通过 `pygame.key.get_pressed()` 进行轮询式按键状态检测。需暴露 `is_boost_pressed() -> bool` 或等效接口供游戏逻辑调用。
- **IP-003 game.py / 游戏主逻辑**: 在游戏主循环的 tick 计算环节，根据加速状态动态调整 tick 间隔（`current_interval = BASE_TICK_INTERVAL / current_multiplier`）。在状态转换（pause, game_over, restart）时触发加速状态复位。在非游玩状态下忽略加速输入。
- **IP-004 renderer.py / 渲染层**: 接收加速状态标志（`is_boosting: bool`），在每帧渲染时根据该标志绘制视觉指示（蛇身颜色修改、HUD 叠加文本、或边框效果）。
- **IP-005 main.py / 游戏主循环**: 窗口焦点事件（WINDOWEVENT_FOCUS_LOST / WINDOWEVENT_FOCUS_GAINED）的监听，用于触发加速状态清除。事件分发顺序需确保焦点事件在游戏逻辑更新之前处理。
